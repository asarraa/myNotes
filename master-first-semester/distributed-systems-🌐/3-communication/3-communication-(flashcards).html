Communication

Q: What is the role of the Physical Layer in the OSI model?
A: It describes how bits are transmitted over a physical medium between two directly connected devices.

Q: What does the Data Link Layer do in networking?
A: It defines how bits are grouped into frames, handling error control and flow control between directly connected nodes.

Q: Which OSI layer is responsible for routing packets across a network of computers?
A: The Network Layer.

Q: What is the primary function of the Transport Layer?
A: It manages the end-to-end transmission of data between two nodes, independent of the lower layers.

Q: Name two standard transport layer protocols and their key differences.
A: - TCP: Reliable, connection-oriented, stream-oriented.

  • UDP: Unreliable, connectionless, best-effort datagram transmission.

Q: What is TCP and what is it used for?
A: TCP (Transmission Control Protocol) is a reliable, connection-oriented protocol used for ensuring ordered and error-free communication, like in file transfers and web browsing.

Q: What is UDP and when is it used?
A: UDP (User Datagram Protocol) is an unreliable, connectionless protocol that provides faster communication, used in real-time applications like video streaming or online gaming.

Q: How do modern networks handle the higher OSI layers (5-7) in practice?
A: These layers are often merged together, with middleware providing common services for applications.

Q: What does middleware handle in distributed systems?
A: Middleware provides services like data marshaling, naming protocols, security protocols, and scaling mechanisms (replication and caching).

Q: What are application-specific protocols?
A: These are protocols tailored for specific applications, such as HTTP for web traffic or FTP for file transfers.

Q: What is data marshaling/unmarshaling in distributed systems?
A: The process of converting data structures into a transmittable format and vice versa for integration between systems.

Q: What does the Security Protocol in middleware do?
A: It ensures secure communication through encryption, authentication, and integrity checks.

Q: Which types of communication are offered by middlewares?
A: Transient (decoupling sender and receiver in space) vs Persistent (decoupling in time ad space) and Synchronous (the invoker waits for the invokee) vs asynchronous (the invoker goes on in parallel after invoking the invokee)

Remote procedure call (RPC)

Q: what's the idea behind RPC?
A: a middleware that allows a client to invoke a procedure on a server in such a way that the client stops executing while the server processes the procedure. Once the server completes the execution, the client resumes its operation, as it would in traditional procedural programming languages.

Q: Is RPC message based?
A: Yes, In RPC, when a client invokes a procedure on a remote server, the request is encapsulated in a message that is sent over the network. This message contains the necessary information for the server to execute the requested procedure, including the procedure name, parameters, and any context required for execution.

Q: What are the different mechanism to pass parameters to a procedure?
A: the following

  • By value. Like in C when passing basic data types
  • By reference. Like in C when passing arrays or in Java when passing objects
  • By copy/restore. At first they are passed by value, but then at the end of the execution the new values are copied back to the invoker.

Q: In RPC, how is it possible to do procedure calls that can be transmitted and then reinterpreted on the server side, which  might be running on a different machine, using a different CPU architecture, or written in a different programming language?
A: This process of converting the procedure call into a format that can be transmitted and then reinterpreted on the server side is called serialization and marshalling. Serialization refers to converting data into a byte stream, while marshalling involves packaging the procedure call (including parameters) in a way that allows it to be re-executed on the server (es: little endian vs big endian).

Q: How are serialization and marshalling achieved? What do the IDL has to do with those?
A: Middleware provide automated support to achieve serialization and marshalling, this is done by using an Interface definition language (IDL), which is a data representation format to be used during communication.

Q: What is the IDL in the context of remote procedure calls? Describe it
A: Interface definition language (IDL) is a data representation format to be used during communication. It represents the interface of a procedure call, separating the service interface from its implementation. By providing mapping onto target languages, it raises the level of abstraction of the service definition, making the service definition language independent.

Q: What is Sun RPC primary use?
A: Sun RPC is the de facto standard for remote procedure call.

Q: What data format does Sun RPC use for serialization?
A: Sun RPC uses XDR (eXternal Data Representation) for data serialization.

Q: What transport protocols can Sun RPC utilize?
A: Sun RPC can utilize either TCP or UDP for transport.

Q: What parameter passing method is allowed in Sun RPC?
A: Sun RPC only allows pass by copy, with only one input and one output parameter.

Q: What additional services does DCE (Distributed Computing Environment) provide beyond basic RPC?
A: In addition to RPC, DCE provides a comprehensive framework, including:

  • Directory services for locating resources across the network
  • Distributed time service to synchronize time across systems
  • Distributed file service for file sharing across multiple systems
  • Security services based on Kerberos for secure communication and authentication.

Q: What is the main problem when binding a client to a server in RPC?
A: The client needs to find the server's location (IP) and establish communication without hard-coding this information.

Q: How does Sun RPC's portmap solve the client-server binding issue?
A: The server registers its service and port with portmap on a well-known port (111). The client queries portmap to find the service's port, then connects directly to the server.

Q: How does a client find the portmap service in Sun RPC?
A: The client must know the server's IP and contacts the portmap daemon on port 111 to retrieve the actual service port.

Q: What problem does the DCE directory service solve?
A: It provides location transparency, allowing clients to find servers without knowing their IP address or port in advance by querying a central or distributed directory service.

Q: How does a client locate a DCE directory service?
A: The client either knows the directory service's well-known IP/port or uses a distributed service for scalable queries across multiple servers.

Q: What is dynamic activation in RPC, and how is it implemented in Sun RPC?
A: Dynamic activation launches server processes only when a client makes a request. In Sun RPC, the inetd daemon starts the server when needed to conserve resources.

Q: What is Lightweight RPC?
A: It's a form of RPC which works among processes in the same machine (es: cut and past in windows), remember that processes don't share mamory

Q: What is Asynchronous RPC? How is it achieved?
A: Is a way to make RPC asynchronous, so to avoid the sender for waiting for the receiver to return:

  • If no result from the procedure call is needed, the execution from the sender can resume after an acknowledgment is received from the server.
  • If a result is needed, the called may return immediately with an acknowledgement (as above) and then may (asynchronously) invoke the caller back after the result is ready

Q: What is the difference between batched and queued RPC in Sun RPC, and how does it enhance communication, especially for mobile hosts?
A: Batched RPC in Sun RPC allows clients to buffer multiple RPC calls without results, sending them together when a non-batched call is made or when a timeout occurs, allowing for asynchronous communication. Queued RPC involves storing requests and managing their delivery, which can be crucial for mobile hosts that may disconnect; the server periodically attempts to deliver replies when the host reconnects

Remote method invocation

Q: What's the idea of remote method invocation?
A: The idea is offering the ability to invoke operations throught the network. In RPC we assumed to live "in a procedural world", so every language were procedural. Now we assume to live in a object oriented world. We would like now to be able to invoke methods from remote objects.

Q: How are interfaces shared in RMI?
A: For sharing interfaces in RMI, the Interface Description Language for distributed objects is used, it is a much richer version of the IDL used in RPC.

Q: Which are the two major examples of RMI middlewares?
A: Java RMI and OMG CORBA. The first is in Java, the former is multilanguage.

Q: How do proxy work in RMI?
A: Proxy on the Client shows the same interface as the one of the object. Whe Client invokes that method, proxy serializes the parameters and send them throught the network, at the destination the skeleton will deserialize parameters an ivokes the method on the actual object.

Q: How is passing by reference achieved in RMI?
A: We can use a proxy to the object, which is an object with the same interface as the object and that "refers" to: the remote object. Meaning methods called on that proxy will result in invoking methods on the original remote object.

Message oriented communication

Q: What's the idea of message oriented communication? What are the main characteristics?
A: Centered around the notion of one-way message/event.

  • Less coupling between the sender and the receiver of the message, compared to the others.
  • Usually asynchronous
  • Often supporting persistent communication
  • Often supporting multi-point interaction

Q: What is message passing in message-oriented communication?
A: Message passing is the simplest form of communication, typically provided by the network OS, where processes communicate by sending and receiving messages directly.

Q: What is MPI (Message Passing Interface)? How does it work?
A: MPI is a middleware-provided method for message passing, commonly used in distributed computing environments to enable efficient communication between multiple nodes. It provides different services besides pure read and write.
Communication takes place within a known group of processes: each process within a group is assigned a local id, the pair (groupID, processID) represents a source or destination address.

Q: Can MPI send messages in broadcast? If so, which forms of broadcasting are there?
A: Yes, messages can also be sent in broadcast to the entire group (MPI_Bcast, MPI_Reduce, MPI_Scatter, MPI_Gather)

Q: What is an overlay network, and how is it used in middleware?
A: An overlay network is a virtual network built on top of the physical network. It operates on top of an existing network infrastructure (like the Internet) and provides additional features and functionalities by routing messages based on application-level criteria instead of relying on traditional network protocols or details like IP addresses.
It is used to manage communication and routing efficiently, often supported by communication servers in systems like message queuing and publish/subscribe.

The overlay consists of nodes (also called relays or brokers) that communicate with each other and manage message routing based on higher-level criteria. These nodes can be distributed across different locations in the underlying network.

Q: What is a Berkeley Socket? How are they identified?
A: programming interface used for inter-process communication (IPC). That is an abstraction that allows programmers to use message passing.

Q: What is a multicast socket?
A: Multicast sockets are a special type of network socket used to send messages to a group of recipients simultaneously in a multicast group, which can be open or closed. A multicast group is identified by a multicast IP address. A sender sends data to a multicast IP address, and all devices interested in this group receive the data. Multicast generally uses the UDP (User Datagram Protocol), because it doesnâ€™t require a connection and can efficiently broadcast data to multiple receivers. In this setup the port number's role is primarily to specify which application or process on a receiving host should handle the incoming data, rather than identifying the host itself.

Q: What does it mean that sockets are protocol independent?
A: When we say that sockets are protocol-independent, it means that the socket API does not tie you to a specific network protocol, such as TCP or UDP. Instead, it provides a generic interface that can be used with different network protocols, allowing flexibility in how communication is performed

Message queuing

Q: What is message queuing in middleware?
A: Message queuing allows messages to be stored in a queue and processed later, enabling asynchronous communication where the sender and receiver do not need to be connected at the same time.

Q: Why is message queuing intrinsically peer-to-peer?
A: Because is a point-to-point communication where every node has a in going queue and outgoing queue, so no distinction between the two. However, message queuing can be used to design an client-server architecture by making the node "act" as the server.

Q: How are queues identified in a message queuing system?
A: Queues are identified by symbolic names (human-readable names) rather than raw network addresses.

Q: What service is needed to convert symbolic queue names into network addresses?
A: A lookup service, possibly distributed, is required to convert symbolic queue-level addresses into network addresses.

Q: What is an overlay network in message queuing?
A: Overlay networks allows message brokers to manage queues and ensure reliable message delivery among producers and consumers.

Q: How does an overlay network improve message queuing systems?
A: It improves fault tolerance and provides multi-point communication (similar to multicast) without requiring IP-level multicast support.

Q: What is the function of a message broker in message queuing?
A: A message broker acts as a gateway, performing message conversion between different formats, which helps integrate various subsystems. N.B. The broker is not associated to a single queue but acts on all the queues based on shared conversion rules.

Q: What are the main characteristic of message queuing communication?
A: Point-to-point, persistent, asynchronous

Publish-subscribe model

Q: What is the publish/subscribe model in middleware?
A: In the publish/subscribe model, a publisher sends messages to any subscribers interested in specific message types, allowing multiple clients to receive messages when they are published.

Q: What is the role of the event dispatcher in a publish-subscribe system?
A: The event dispatcher collects subscriptions and routes events to all matching subscribers. It can be centralized or distributed.

Q: How is communication in a publish-subscribe system (characteristics)?
A: Communication is asynchronous, implicit (not direct knowledge of each other's identity or existence), and multipoint, meaning one event can be sent to many subscribers without direct connections between components.

Q: What is the benefit of decoupling components in publish-subscribe systems?
A: Decoupling makes it easy to add or remove components, making the system ideal for dynamic environments.

Q: What is a subject-based subscription?
A: A subject-based subscription allows subscribers to subscribe to predefined topics or categories (e.g., all events about "Distributed Systems").

Q: What is a content-based subscription?
A: A content-based subscription uses filters to specify detailed criteria based on specific content or attributes (e.g., events about "Distributed Systems" with a specific date and location).

Q: How can a publish-subscribe system scale using distributed dispatchers?
A: A distributed dispatcher uses message brokers in an overlay network to route events, allowing the system to scale and improve fault tolerance. A broker acts as gateways in the message queuing architecture, enabling message conversion between different systems or formats. This is useful when you need to integrate multiple subsystems that may use different protocols or message formats.

Q: What is the tradeoff in using a more expressive subscription language?
A: A more expressive subscription language allows finer event filtering but increases the complexity of the implementation.

Q: What kind of overlay network structures and message forwarding policies are there in the publish-subscribe model?
A: the following

  • Acyclic topology
    • Message forwarding: Every broker stores only subscriptions coming from directly connected clients, messages are forwarded from broker to broker and delivered to clients only if they are subscribed. Each subscription table (in each dispatcher) is small as only contains information about directly connected clients, so this process is not so costly in term of subscription, but it is costly on the network
    • Subscription forwarding: Every broker forwards subscriptions to the others, mssages follow the routes laid by subscriptions. In terms of subscription forwarding it is costy, but less costy when publishing (cause message only follow one route).
    • Hierarchical forwarding: Hierarchical forwarding uses a rooted overlay tree structure where messages and subscriptions are routed through dispatchers toward the root. When a client subscribes to an event, the subscription flows upward to the root dispatcher. When an event is published, it is sent to the root, which forwards it downwards only if there are matching subscriptions along that route, ensuring efficient message delivery to interested subscribers.

Stream-oriented communication

Q: What is stream-oriented communication? What kind of sub-streams are there?
A: A communication involving a data stream, which is a sequence of data units.
Stream types: Simple vs. complex streams (composed of sub-streams).

Q: Explain asynchronous, synchronous and isochronous transmission modes in stream-oriented communication
A: here

  • Asynchronous: The data items in a stream are transmitted one after the other without any further timing constraints (apart ordering)
  • Synchronous: There is a max end-to-end delay for each unit in the data stream
  • Isochronous: There is max and a min end-to-end delay (bounded jitter)

Q: What are QoS in stream oriented communication?
A: Quality of Service requirements are non-functional requirements like:

  • required bit rate
  • maximum delay to setup session
  • maximum end-to-end delay
  • maximum variance in delay (jitter)

Q: What is the role of the Differentiated Services (DiffServ) architecture?
A: DiffServ provides mechanisms for offering different levels of service to network traffic (QoS), addressing the limitations of traditional best-effort IP. However, it is not necessarily supported by internet routers. So QoS are usually enforced at the application layer.

Q: How are QoS enforced at the application layer in streaming oriented communication?
A: through

  • Buffering: tries to resolve the variation on the speed of the network by creating a buffer, sacrificing session setup time.
  • Forward/backward error correction: you reach a state which is incorrect one, two solutions:
    • go back to a previous state: backward correction, not the best approach for this kind of service
    • try to go to the right state: forward correction. Es: in the case of videos, you have the main frame and then deltas from the previous and the next, if you lost a delta, you can try to reconstruct it from the next deltas
  • Interleaving data: Data interleaving involves rearranging the order of data elements to disperse them across different storage or transmission channels. This technique is especially useful for managing error correction and ensuring that data can be reconstructed accurately even in the presence of errors.
