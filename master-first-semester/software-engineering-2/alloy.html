Structural design

Signature declaration

To declare a top-level signature we use the keyword sig followed by the signature name and a list of field declarations between braces
sig Object{}

Everything is a relation

Signatures define sets, which are relations of arity 1.

To declare a subset signature, the keyword in should be used after the signature name, followed by the name of the including (or parent) signature. However the keywords Extends and abstract can help us designing subsets.

Multiplicity constraint

It is possible to insert multiplicity constraint that restrict the cardinality of the signature:

  • some
  • lone
  • one
    (see Multiplicity keywords)
Singleton

A singleton is a set with multiplicity one

Multiple declaration

It is possible to declare many signatures, that share the same features (multiplicity, eventual parent signature, etc), they can be defined all together in the same declaration.
one sig Red, Blue, Green extends Color{}

Enum declaration

The enum keyword allows to define multiple signature all together, imposing a order on them. It can be applied when:

  • the multiplicity of the signatures is one
  • all the signatures extend the same top level signature

es: enum Color { Red, Green, Blue} this is almost equivalent to one sig Red, Blue, Green extends Color{}, the only difference is that the first imposes a total order between the respective atoms, corresponding to the order in which they are declared inside the braces.

extends and abstract

extends is used to guarantee that subsets of the same set are disjointed.

#example
sig File extends Object{}
sig Dir extends Object{}

#guarantees the following
fact{no File & Dir}

abstract signature guarantees that extension signatures form a partition of the parent signature.

#example
abstract sig Object{}
sig File extends Object{}
sig Dir extends Object{}

#guarantees the following
fact{Object = File + Dir}

Field (relations) declaration

Fields are essentially mathematical relations that connect atoms to other atoms.
A field declaration consists of its name followed by a colon (:) and the target set, optionally preceded be a multiplicity declaration:

  • set
  • some
  • lone
  • `one
    (see Multiplicity keywords)
Default

Default multiplicity for fields : one

Example
 sig Dir extends Object {
	  entries : set Entry,
	  <rel2> : [multiplicity] <Signature>
  }

Restrict multiplicities of a binary relation

Special syntax to restrict multiplicities of both end points of a binary relationship:
To state that a relation r relates every atom of source signature A to at most one atom of target signature B, and every atom of B to at least one atom of A, we could write the formula r in A some -> lone B.

Default

Default multiplicity for a non explicit end point multiplicity: set

Relational constants

Constants

none: denotes the empty set
univ: the set of all atoms in all signatures
iden: the binary relation that associate each atom in univ to itself

Mutable signatures

When there are mutable top-level signatures (declared with var), both univ and iden are mutable also, with the former containing, in each state, all atoms contained in all signatures at that state.

Relational operators

Relational operators

The following is the set of relational operators that can be used to combine relations:
&: intersection between conditions.
or: union between condition
implies[no]: logic implication [or not implication]
iff: double implication
+: set union
-: set difference
->: Cartesian product
no: checks for set emptiness.
lone: checks for the set to contain at most one atom
some: true if set is non-empty
all: universal quantification.

disj: can be added between quantifier and variables to tell the variables are different.

.: Composition
^: Transitive closure. Given a relation r, ^r is the same as r + r.r + r.r.r + ... (at least one step)
*: Reflexive transitive closure. Is similar to the ^ but also involves the identity relation: * = ^ + iden (zero or more steps)
~: Transposition. Given a BINARY relation r, ~r is the relation which is composed by all the pairs in the r relation but with inverted elements. Used to express that a particular relationship is reflective: fact { wife = ~husband }

On none and no

none is a unary relation, so it cannot be "compared" to any kind of relation but only to unary ones (sets). For using it with relations of different arity, we will need to "create" the relative empty relation, through the Cartesian product operator.

none for empty binary relation

The empty binary relation is denoted by the Cartesian product none -> none: this expression denotes the empty set like none, but for the type system it is a binary relation, so writing R = (none -> none) does not raise a type error.

Better use no

Better use the no operator to check if a generic relation is empty. Since it is a cardinality check keyword, it can be used with relations of arbitrary arity.

Filtering a relation

The <: or :> can be used to restrict the domain of a relation, based on the type of the elements involved.

The following restrict the domain of relation to only the entries whose first element belongs to A.

A <: relation

The following restrict the domain of relation to only the entries whose second element belongs to A.

relation :> A
Tip

Notice that this can be done also using the Cartesian product operator ->:

relation & (A - > B)

This kind of operations are particular useful when dealing with Relational constants.

Example

For example we could use the iden constant to check if a particular relation R (defined on a signature A) is reflexive. We start from the iden relation and filter for the couples whose first element is A:

(A <: iden) in R

Dynamic operators

Info

always: at all time instants from now on
historically: all instants before the current one.

eventually: one instant among the current one and all the future ones
once: one instant among the previous ones (past version of eventually)

before: the previous instant
after: the successive instant

Combined operators
eventually always: all instant after a certain instant in the future (or the current one)

On composition

The composition operator (.) in Alloy is a fundamental tool for navigating relationships between atoms. It is similar to accessing attributes in object-oriented programming. Here’s a summary of its usage and properties:

  1. Basic Usage:
    • The composition operator links sets and relations to retrieve related atoms.
    • Example: If e is an Entry, e.name gives the name of e, a singleton set since name has a multiplicity of one.
  2. Non-Singleton Sets:
    • The operator works with arbitrary sets and relations of varying multiplicities.
    • Example: d.entries retrieves all entries in directory d. For all directories, Dir.entries gives all their entries combined.
  3. Bidirectional Navigation:
    • Relations can be navigated forward (from source to target signature) or backward (from target to source signature).
    • Example: entries.e retrieves all directories containing e, while entries.Entry gives all directories containing any entry.
  4. Chaining Compositions:
    • Compositions can be chained to explore complex relationships.
    • Example: d.entries.object retrieves all file system objects in the entries of directory d.
    • Backward navigation can also be chained, such as entries.object.d to find directories containing specific objects.

Facts

fact optionalname {
	signatures | conditions
}

#example 
fact {
  no File & Dir
  Object = File + Dir
}

fact {
  // Different entries in the same directory must have different names
  all d : Dir, disj x, y : d.entries | x.name != y.name
}


fact {
  // Different entries in the same directory must have different names
  all d : Dir, n : Name | lone (d.entries & name.n)
}

N.B. constraints in different lines inside a fact are implicitly conjoined (&).

Running

Run <nameOfSpecificaiton> {}returns instances of the specification. The instances will satisfy all the specified constraint.
It is possible to add a constraint directly into the parenthesis. Empty brackets means true.

With New we can iterate over the non symmetric (that differ only in the naming of atoms) instances returned by a run.

Run scope

A run is executed with a scope for each signature: maximum number of atoms that can inhabit the signature.
run example{} for <globalScope>

Example

run example {} for 4

Default

Default scope for top level signatures = 3. If a personalized scope is not provided.

But and exactly keywords

Also possible to set a different scope for each signature, by adding the following in the run line:
but <scope1> <signature1>, <scope2> <signature2>, ... <scopeN> <signatureN>
es: run example {} for 4 but 2 Entry

Note

The but only works after a for

Also possible to set the exact number of atoms that inhabit a certain signature by adding the following:
exactly <ExactScope1> <ExactSignature1>, <ExactScope2> <ExactSignature2>, ... <ExactScopeN> <ExactSignatureN>

Complete example

run example {} for 4 but 2 Entry, exactly 3 Name
run example {} for 3 Node, exactly 3 Id
run example {} for Exaclty 3 Node, exactly 3 Id

Note

The exactly only works after a for ....
To be precise, exactly can only be used if every signature is quantified

Scope and multiplicity

Multiplicity constraint on signatures declarations also affect the scope: error if an incompatible scope is set to a signature with a multiplicity constraint.

Parent and extentions signatures scopes

If the scope of extension signatures enforces a minimum number of atoms, the scope defined for the parent signature will automatically grow to accommodate that minimum number.
Es: in the following command the scope of Object will automatically grow to 4 since Object must contain at least the root and three files.
run example {} for 3 but 2 Object, exactly 3 File

Steps - dynamic analysis

In case of a dynamic specification (there are mutable relations), is it possible to specify the number of instants we want to represent.

Example

run show for 3 but 1 Book, 3 steps

Assertions

Assertions are used to find counterexamples which are the instances of a model.

assert NoSelfFather 
	{ no m: Man | m = m.father } 
	
check NoSelfFather

N.B. check command checks that an assertion is implied by the conjunction of all declared facts. It is also possible to directly insert the body of the assert directly into the check: check { no File & Dir }

Default

The default number of top level signatures is 3
The default number of maximum number of different transitions is 10

Personalized scope

check AtMostOneLeader 4 but 20 steps

Function and predicates

Functions and predicates are reusable expression. The only difference is that functions also return a value.

Predicate example
pred ownGrandPa [p: Person] {
	p in p.grandpas
}
Function example
fun grandpas [p: Person] : set Person {
	p.(mother+father).father
}
# this function returns a set of Persons

Facts vs predicates

They are both used to express constraints but

  • Facts must hold globally for any atom and relation in the model
  • Predicates have to be invoked on specific instances

Mutable relations and sets

Old style

Consider the following example:

sig Name, Addr {}
sig Book {
addr: Name -> lone Addr
}

Since the addr relation is unmutable, the evolution of the address book is done by means of additions and deletions required to explicitly represent the address book before and after the operation:

pred add [b, bpost: Book, n: Name, a: Addr] {
bpost.addr = b.addr + n -> a }

Dynamic specification

In alloy is possible to allow relation to change over time, by means of the keyword var:

sig Book {
var addr: Name -> lone Addr
}

In this way, different instants in time can be represented. There are different ways:

  • after keyword, which means "at the next instant" 
    pred show [b: Book] {
	#b.addr = 0
	after ( #b.addr = 1
		and
		after #b.addr = 2 )
}
  • ; keyword, which is the same as after
pred show [b: Book]{
		#b.addr=0; #b.addr=1; #b.addr=2
}
  • ' which can be concatenated to an object and means "the next value of the object"
pred add [b: Book, n: Name, a: Addr] {
b.addr' = b.addr + n -> a
}

Multiplicity keywords

Can be used insignatures cardinality and field declaration and multiplicity restriction

Multiplicity keywords
  • set: zero or more
  • some: at lest one
  • lone: at most one
  • one : exactly one

this keyword

The keyword this can be used in the following situations:

  • In declarations and signature facts, this acts as a variable implicitly bound to each instance of the signature. So a declaration of the form 
     sig Foo { ... } { copacetic[this] }
    is equivalent to 
      sig Foo { ... }
  fact { all f : Foo | copacetic[f] }

Set comprehension

Useful for defining return values with formulas


fun suggestRecipe[u : User] : set Recipe{
	{r : Recipe | (some i : r.ingredients | i in u.has)}
}
