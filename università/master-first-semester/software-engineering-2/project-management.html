Illustrating project schedule

  • Gnatt charts are used for illustrate project schedule. They are the result of one of the following methods:
    • Work Breakdown Structure
    • Precedence Diagram Method
    • Critical Path Method
Work Breakdown Structure (WBS)

Breaks down the work that must be done into tasks that can be easy to estimate, assign, track.

Precedence Diagram Method (PDM)

Technique used for constructing a schedule starting from the precedence relationships between activities, defining which task (predecessor) triggers the other (successor) by using the following types of dependencies:

  • finish to start: the finish of the predecessor triggers the start of the successor
  • start to start
  • finish to finish
  • start to finish

LAG time can be added to dependencies in order to introduce a positive or negative delay.

Critical Path Method (CPM)

Critical path method is used to estimate the minimum project duration.
Given a project schedule network diagram and an estimation of activities
duration, calculate the early start, early finish, late start, and late finish dates that do not delay the project finish date or any schedule constraint.

Risk management plan

Define a framework to identify, analyze, treat and monitor the risk constantly.

Definition

Risk: Potential problem or threat

Identification can be based on:

  • Prior experience
  • Brainstorming
  • Lessons learned from similar project
  • Checklists
    • Rule of thumb: pose each risk in the form "if situation, then consequence, for stakeholder".
      Measures of risk are:
      • Likelihood: expressed as probability
      • Impact: depends on the nature of the event
Executing, monitoring and controlling
Execution
  • Launch the project : kick off meeting
  • Acquire and manage project team (internal and external resources)
  • Acquire the required equipment and materials and external services
  • Perform the work identified in the schedule
  • Perform monitoring and controlling activities
Monitoring

Monitoring: consists of collecting data about where the project
stands, since projects never stick to the initial plans due to changes,
problems, etc.

  • Gather up-to-date data
  • Update the schedule
  • Compare actual schedule with baseline schedule
  • Compare actual costs to assigned budget

Deal with risks: In case risks become reality, follow the mitigation strategy defined in the risk management plan and revise the mitigation strategy if needed.

Earned value analysis
It is a methodology to assess project performance and progress.

First, let's understand the basic variables:

  • (Budget at Completion) - The total money you planned to spend on the project
  • (Planned Value) - How much money you planned to have spent by now
  • (Earned Value) - The value of work actually completed, based on your original budget
  • (Actual Cost) - The real money you've spent so far

Schedule Formulas:

  1. Schedule Variance:

    • Interpretation: Shows if you're ahead or behind schedule in dollar terms
    • If positive: you're ahead of schedule
    • If negative: you're behind schedule

  2. Schedule Performance Index:

    • Interpretation: Shows how efficiently you're using time
    • If > 1: you're working faster than planned
    • If < 1: you're working slower than planned
    • Example: SPI of 0.8 means you're only completing 80% of the work you planned to complete

Cost Formulas:

  1. Cost Variance:

    • Interpretation: Shows if you're over or under budget
    • If positive: you're under budget
    • If negative: you're over budget

  2. Cost Performance Index:

    • Interpretation: Shows how efficiently you're using money
    • If > 1: you're spending less than planned
    • If < 1: you're spending more than planned
    • Example: CPI of 0.9 means you're getting only 90 cents of value for each dollar spent

Estimate at Completion (EAC) Formulas:

  1. (Same CPI assumption)

    • Interpretation: "If we keep spending money at the same efficiency rate as now, this is what the final cost will be"
    • Uses past cost performance to predict future costs

  2. (Baseline rate assumption)

    • Interpretation: "If we forget our past performance and spend exactly as originally budgeted for the remaining work"
    • More optimistic approach, assumes past problems won't affect future work

  3. (Same CPI and SPI assumption)

    • Interpretation: "If both our time and cost efficiency stay the same as they are now"
    • Most comprehensive as it considers both schedule and cost performance
    • Usually gives the most pessimistic (but often realistic) estimate
Controlling

Controlling is where you implement corrections to get your project
back on track. It means balancing scope, time, cost, resources and quality.

  • If schedule is important --> fast-tracking and crashing
  • If money is a priority --> reduce resources or overhead costs
  • If schedule, money, resources are non negotiable --> reduce scope eliminating tasks associated with it

Fast tracking: push tasks to start earlier than they would by introducing a
negative lag time, if needed.

  • This can be done only if the considered tasks can actually be parallelized.
  • No cost increase but higher risk.

Crashing: shorted the tasks on the critical path by adding new resources.

  • Feasible if addition of resources is actually beneficial
  • It implies a cost increase.

Dimensional and functional metrics: LOC and FP

From the product model, one can estimate the size of the software and derive the following metrics: FP (function point), LOC (lines of code).

Estimate software size

  • Option 1:
    • From the RASD or other informal preliminary documents, identify the main characteristics of the software to be and classify them in terms of Function Points
    • From the number of Function Points determine the software LOC estimation
  • Option 2: Estimate LOC directly, based on previous experience/projects.
Function Points

Basic assumption: the size of software can be characterized based on the "functions" that it has to offer.

Based on a combination of program characteristics, function points can vary, in the proposed method the identified functions are:

  • Internal Logic File (ILF): homogeneous set of data used and managed by the application
  • External Interface File (EIF): homogeneous set of data used by the application but generated and maintained by other applications
  • External Input: Elementary operation to elaborate data coming from the external environment
  • External Output: Elementary operation that generates data for the external environment
  • External Inquiry: Elementary operation that involves input and output (without significant elaboration of data from logic files)
