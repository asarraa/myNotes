Update definition

An update happens only when a resource gets its value rewritten based on its previous value, thus a transaction must read a value before writing it. In case a transaction doesn't perform the reading of a resource before writing it, then it is not an update. Ex: is a lost update, while is not , since doesn't read , just over-write it.

From conflict graph to serial schedule

Conflict graph acyclic schedule view serializable. To find the equivalent serial schedule, find a topological order of the nodes of the graph, that order will be the order of transactions in the serial schedule.
N.B. If the topological order is not unique, this means that multiple equivalent schedules exist.

Classify schedule (CSR and VSR)

How to proceed:
- check if schedule is in CSR
- If it is, then it is also in VSR
- otherwise, look for a condradiction:
- If you find one, then it is not in VSR
- If you don't find one, we suspect it is VSR, to find the serial schedule we can proceed
trying moving inactive reads, otherwise we need to list all the constraint (relative
to final writes and read from), doing the graph and trying to find a schedule.

Locking

Locking normale non elimina nulla.
2PL elimina tutto tranne dirty reads (che richiede di gestire aborts) e phantom insert nel caso in cui l'elemento che viene inserito sia diverso da quello loccato da T1
2PL stricts elimina anche dirty read (gestisce aborti facendo rilasciare i lock solo dolo commit/abort

Update lock e Hierarchical lock cercano di diminuire la frequenza dei deadlock, il primo evitando i deadlock sul pattern update (i più comuni) e il secondo evitando di diminuire troppo la concorrenza prendendo lock su intere tabelle quando non necessario.

SL and UL

SL and UL can cohabit: if a resources is under UL, it can be shared-locked, however before the upgrade of the UL, the transaction which has shared-locked it must release it.

Fagin

  • I sorted accesses vanno fatti su ogni lista finché non si trovano k elementi che compaiano in tutte le liste. Ciò vuol dire che anche se concludo la ricerca degli elementi comuni nella riga , devo completare i sorted accesses per quella riga.
  • La scoring function si calcola (e gli eventuali random accesses vanno fatti) per tutti gli elementi visti durante la ricerca dei k elementi comuni, non solo per questi ultimi.

NRA

Si effettuano sorted accesses riga per riga e si mantengono LB e UB per ogni elemento.
Intuitivamente il LB è calcolato usando i dati che abbiamo già visto completati eventualmente il minimo possibile (0.0).
L'upper bound è calcolato usando i dati che abbiamo già visto completati con il massimo possibile, ovvero con il dato più basso visto nella colonna dell'attributo mancante.
Creiamo il rank ordinato per i lower bound

Bisogna stopparsi quando il k'esimo elemento non può più essere battuto, ovvero quando gli upper bound degli elementi sotto di lui sono inferiori (o uguali) al suo lower bound e anche un eventuale nuovo elemento (Rappresentato dal t) ha valore della scoring function minore (O uguale) al lower bound del k'esimo elemento.

Skyline

  • Before performing the SFS (sort filter skyline) algorithm, sort the elements using:
    • Sort attribute after attribute in order not to have ties that may lead to the violation of the topological principles for which an element t1 cannot be dominated by an element that follows it in the ranking.
    • SOME monotone scoring function involving all attributes.
  • If a tuple is dominated by at least one point of the window, then it will be not included in the skyline.

Hashed Join

Condition: both tables are hashed on the same key (join) attribute.

Working: Find the first (or next) key value v in the left table, we can directly use this key to look up matching records in the right table using the hash function. If the left and right table have matching tuples for v, those tuples are combined (joined) and added to the result set. The process continues for each key value in the left table, retrieving and joining matching records from the right table.
