Peer-to-Peer architecture

The Peer-to-Peer (P2P) architecture is a distributed system design paradigm where there is no distinction between servers and clients. In a P2P network, every node (or peer) can act both as a service provider and a service consumer, making the system more decentralized and scalable.

One key advantage of P2P systems over traditional client-server architectures is scalability. In a P2P system, as more users (service consumers) join the network, they also contribute resources (service providers), increasing the systemâ€™s overall capacity.

Key Characteristics of P2P Systems

  • Independent Nodes: Each peer functions independently from the others.
  • Dynamic Participation: Nodes are not permanently connected; they can join and leave the network unpredictably.
  • Variable Capabilities: The resources available at each node (such as bandwidth, storage, and processing power) can vary significantly between peers.
  • Large-scale Operation: P2P systems can scale up to global, internet-wide networks.

Fundamental Issue: Resource Retrieval

One of the major challenges in P2P systems is efficiently retrieving resources across the network. There are two primary types of retrieval operations:

  1. Search: Finding documents or resources of a particular type within the network.
  2. Lookup: Locating a specific item or exact copy of a file.

Centralized Search in P2P: Napster

Napster was the first widely adopted P2P file-sharing application. Its core concept was to allow users to share their storage and bandwidth to distribute files. However, Napster employed a partially centralized model:

  • Join: When a user joins the network, they must register through a central server.
  • Publish: Users publish lists of files they wish to share via the central server.
  • Search: The central server maintains a searchable index of files and the users who possess them. Searches are handled centrally, and users query the server to find out who holds the desired file.
  • Fetch: Once a file is located, the actual download (fetching) occurs directly between peers, bypassing the central server.

Pros:

  • Simplicity: Napsterâ€™s design was straightforward and easy to implement.
  • Efficient Search: The search process was fast and efficient (scope of ) since the central server handled all queries.

Cons:

  • Single Point of Failure: The central server acted as a bottleneck and a potential single point of failure, making the system vulnerable.
  • Server Load: The central server had to maintain state information for all users () and handle all search queries, putting a heavy load on the server.

Query flooding: Gnutella

In Gnutella, file requests are distributed across the network rather than through a central server. This fully decentralized approach involves each node communicating with its neighbors. The system operates as follows:

  • Join: New nodes join the network by contacting well-known anchor nodes. These anchor nodes can be found in a decentralized manner, such as through internet forums or other publicly available sources. Once a new node wants to join, it sends a PING message to an anchor node. The anchor node forwards this PING message to other nodes in the network, and any nodes with available capacity can reply. A direct connection is then established between the joining node and these responding nodes.
  • Publish: Thereâ€™s no need for explicit publishing of files, as each node simply keeps its own files locally.
  • Search: Gnutella uses a query flooding method for search and lookup. A node sends a search query to its neighbors, which propagate the query to their neighbors, and so on, until the resource is found. The response travels back along the same path to the requesting node.
  • Fetch: Once a resource is located, the file is fetched directly from the peer holding it.

Pros:

  • Fully Decentralized: No central server is needed, and the process of finding anchor nodes is itself decentralized, using resources like internet forums.
  • Distributed Search Costs: The cost of search operations is spread across all nodes rather than being concentrated on a single entity.
  • Flexible Search: Each node searches only its local files, allowing for efficient, localized search strategies.

Cons:

  • Limited Search Scope: The system uses a "hops to live" (TTL) mechanism, which limits the number of nodes a query can pass through. This means thereâ€™s no guarantee that a file will be found, even if it exists in the network.
  • High Search Overhead: Query flooding can lead to excessive message traffic, making searches expensive in terms of bandwidth and processing.
  • Unstable Network: The network can be unstable as nodes frequently join and leave, potentially disrupting connections and search continuity.

Hierarchical Approach: Kazaa

Kazaa operates using a hierarchical P2P model, where special nodes called supernodes provide services. From a regular peerâ€™s perspective, the system appears somewhat centralized, with supernodes acting as central points for resource discovery and management. However, from the supernodes' viewpoint, the network operates in a decentralized manner, as they use a query flooding protocol to communicate with other supernodes within their own smaller network scope.

Hereâ€™s how Kazaa works:

  • Join: When a client joins the network, it contacts a supernode to integrate itself into the system. The supernode acts as the intermediary, managing the clientâ€™s search and resource discovery needs.

  • Publish: Regular peers (clients) send a list of the files they wish to share to a supernode. This supernode then indexes the files for future queries.

  • Search: When a peer wants to find a file, it sends the query to its supernode. Supernodes will then flood the query amongst themselves, searching through their indexed files until the requested resource is found.

  • Fetch: Once the resource is located, the file is fetched directly from one or more peers. Kazaa supports fetching different parts of a file from multiple peers simultaneously, improving download efficiency.

Pros:

  • Node Heterogeneity: Kazaa takes into account the different capabilities of nodes (such as bandwidth, computational resources, and availability). More capable nodes can become supernodes, while less powerful ones act as regular peers.
  • Network Optimization: The overlay network of supernodes can be dynamically adjusted to resemble the actual physical network topology. This allows for optimizations based on factors like geographic proximity, reducing delays between supernodes and the peers they serve.

Cons:

  • Uncertain Search Performance: Despite its more structured approach, there are still no guarantees regarding the search scope (how far the query propagates) or the search time (how long it takes to find a resource).

Collaborative systems: BitTorrent

The key idea behind BitTorrent is not primarily focused on searching for resources, but rather on efficiently retrieving files by distributing the download process across multiple nodes. This innovative approach reduces downloading costs by allowing a client to download different parts of a file from several peers simultaneously, enhancing speed and efficiency.

How It Works:

  1. Torrent File: A user receives an identifier for a specific resource in the form of a torrent file. This torrent file contains several key pieces of metadata:
    • File names
    • File sizes
    • Checksums for each block of the file (files are divided into fixed-size blocks)
    • The tracker address, which helps manage peer connections within the swarm
    • Addresses of peers that possess portions of the file

Terminology:

  • Seed: A peer that has a complete copy of the file and continues to share it with others.
  • Leech: A peer that is in the process of downloading the file and does not yet have the complete copy.
  • Swarm: The collective group of seeds and leeches sharing pieces of a file.
  • Tracker: A centralized server that keeps track of all seeds and peers in a swarm and gathers statistics about file distribution.

File Downloading and Sharing:

While downloading, peers simultaneously upload portions of the file they possess to other peers. The concept is that each node in the network has different pieces of the file, which are exchanged between peers.

Every 20-30 seconds, each peer gathers information about how much data they are receiving from others. Based on this feedback, they can decide whether to maintain certain connections or try to connect to other peers. This system encourages contributions: peers who upload more often tend to enjoy faster download speeds, promoting a balanced exchange of data.

Choosing What to Share:

BitTorrent employs a statistical approach to optimize file sharing. Peers share file fragments based on demand, ensuring that the most needed pieces are distributed first. This game-theory-inspired strategy enhances both efficiency and fault tolerance: if one peer crashes, others are likely to have the necessary pieces.

  • Rarest-first strategy: When determining which segment of a file to download first, peers typically opt for the "rarest" fragment, i.e., the one held by the fewest other peers. This approach ensures that all parts of the file remain evenly distributed, preventing bottlenecks where only a few peers hold critical parts of the file.

How BitTorrent Works:

  • Join: The user contacts a centralized tracker server to obtain a list of peers. When a new peer enters the network, it queries the tracker to obtain a list of peers.
  • Publish: A user can run a tracker server to announce a new torrent file and gather peers.
  • Search: BitTorrent lacks a native search function; users must rely on external methods (e.g., search engines) to find torrent files.
  • Fetch: Files are downloaded in chunks from multiple peers. As peers download, they simultaneously upload the chunks they possess to others.

Pros:

  • Incentivizes Sharing: The protocol encourages peers to share resources (upload bandwidth) in exchange for faster download speeds.
  • Efficient File Distribution: Files are divided into small chunks, which can be downloaded from multiple peers at once, resulting in faster and more resilient downloads.

Cons:

  • Central Tracker Dependency: A centralized tracker is essential for coordinating the swarm and providing peer lists. If the tracker goes down, the swarm may struggle to function unless alternative peer discovery methods are implemented.

Secure storage: Freenet

Freenet was created with the idea of enabling a completely anonymous web for both publishers and consumers. It eliminates the concept of file ownership, meaning that when you share resources, you're not only sharing bandwidth but also a portion of your hard drive.

In Freenet, the network is formed as an overlay that is randomly defined. When you publish content, it is disseminated statistically across the network. Since the data travels through various nodes, it becomes impossible to know the original sender. Having a piece of information in Freenet doesn't mean you are the original owner or creator of it.

As resources become more popular, they automatically spread wider across the network, increasing availability. However, there's no guarantee that any particular resource will remain available indefinitely. Users allocate a portion of their hard drive to Freenet, which stores data temporarily. Over time, less-requested files are discarded to make room for newer content.

Routing and Node Behavior

Freenet nodes have approximate knowledge of what their neighbors store, allowing them to route requests intelligently. When a node receives a request, it forwards it to neighboring nodes that are likely to have the requested content, based on similar file IDs.

Each chunk of information is identified by a hash of the content, which serves as the fileâ€™s identifier. The nodes know which files their neighbors might store, based on these hashed identifiers.

How it Works:

  1. Publishing and Requesting Files: When a file is published or requested, it travels through paths where nodes with similar IDs are clustered. This means similar IDs tend to aggregate in certain regions of the network, making routing more efficient.
  2. Learning Through Requests: If Node A receives a file with ID 7 from Node B, it learns that the region around Node B likely contains other files with IDs similar to 7. Node A stores this information in its routing table for future use.
  3. Routing Table Sharing: The information stored in a nodeâ€™s routing table can be shared with other nodes, helping the network become more efficient at finding files over time.
