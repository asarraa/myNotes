Synchronizing physical clock

source: Synchronization slides

Computer clocks drift, that is, they can sign a different "time". Problem: ensure all machines "see" the same global time. The problem is either:

  • Synchronize all clocks against a single one (accuracy)
  • Synchronize all clocks among themselves (agreement)
    Several protocols have been devised.

Positioning and time: GPS

Basic idea: get an accurate account of time as side effect of GPS, by using four satellites to synchronize receiver time with the one of the satellites, which have an atomic clock.

This systems has limitations in accuracy for different reasons, however GPS receivers can be precise within range of few meters and few tens of nanoseconds.

Cristian's simple algorithm

Brief description
Periodically, each client sends a request to the time server, with the assumption that messages travel fast w.r.t. required time accuracy.

Problems
The major problem is that time might run backwards on client machine.
Minor problem is that it takes a non-zero amount of time to get the message to the server and back.

Berkeley simple algorithm

Server collects the time from all clients, averages it, and then re-transmits the required adjustment

Network Time Protocol (NTP)

NTP is the most used protocol to synchronize machines. It is organized in a tree, where the leaves are users' workstation while top nodes are machines with accurate information from atomic time. Exploiting this structure, there are three synchronization mechanisms:

  • Multicast: servers periodically multicast their time to other computers on the same LAN
  • Procedure-call mode: Similar to Cristian's
  • Symmetric mode

In both Procedure-call and symmetric mode, servers exchange pairs of messages, each bearing timestamps of events about recent messages between the two servers. By calculating the difference of timestamps between the two machines, servers can get an estimate of the offset between clocks and a measure of the accuracy of the estimate.

Presicion: 1ms-50ms over internet, 1ms LAN

Totally ordered multicast

Totally ordered multicast uses Lamport Clocks to assure a global order of messages.

Assumptions: reliable and FIFO links. Non failing nodes.
In fact, in case a message could be lost or corrupted or if a node fails (so it can send acks), the processes may wait forever.

Description:

  • Messages are sent to all group members (including the sender itself). Every message carries its Lamport clock.
  • When a message is received by a process, it is added to its local queue, sorted by the message timestamp (Lamport clock).
  • An acknowledgement regarding the message at the top of the queue is broadcasted to every process (including yourself).
  • Once acks from every process are received it is marked as ack-ed. It means that every process has received the message and has it at the top of its queue, meaning the message can be delivered to the application.

Notice that acks carry a Lamport clock too, meaning they are ordered too

Mutual exclusion

To prevent interference and ensure consistency of resource access, mutual exclusion can be used. That is, one process at the time can access a certain resource.

Guarantees
Mutual exclusion protocols must guarantee the following requirements:

  • Safety property: at most one process may execute in the critical section at a time
  • Liveness property: all request to enter/exit the critical section eventually succeed
  • Optional, priority property: if one request happened-before another, then entry is granted in that order.

Assumptions
Reliable channels and processes

Simple centralized approach

A central server, acting as coordinator, manages the lock using a "token". Resource access request and release obtained with respective messages to the coordinator.
Pros: simple approach
Cons: performance bottleneck and single point of failure

Mutual exclusion with scalar clocks

To request access to a resource:

  • A process multicasts a resource request message m, with timestamp Tm, to all processes (including itself)
  • Upon receipt of m, a process :
    • If it does not hold the resource and it is not interested in holding the resource: sends an acknowledgment to
    • If it holds the resource: puts the requests into a local queue ordered according to (process ids are used to break ties)
    • If it is also interested in holding the resource and has already sent out a requests, compares the timestamp with the timestamp of its own requests. If is the lowest one, sends an acknowledgement to , otherwise it put the request into the local queue above.
  • On releasing the resource, a process acknowledges all the requests queued while using the resource.

A resource is granted to when its request has been acknowledged by all the other processes.

Token ring solution

Processes are logically arranged in a ring, ordered by their process id.

Access is granted by a token that is forwarded along a given direction on the ring:

  • A process not interested in accessing the resource forwards the token
  • Resource access is achieved by retaining the token.
  • Resource release is achieved by forwarding the token.

Guarantees: notice that while the safety and liveness property are respected, the optional property is not.

Conclusion

By comparing the number of messages per entry, the delay before entry and the possible problems of the three proposed solution, we can conclude that in this case the distributed one is not the best one.

Leader election

Many distributed algorithms require a process to act as a coordinator (or some other special role). The problem tackled by these algorithms is that of making everybody agree on a new leader when the old one is no longer available.

Assumptions
Minimal assumptions are required:

  • Nodes are distinguishable (typically by means of identifiers)
  • Processes know each other and their IDs

Guarantees
The non-crashed process with the highest ID at the end of the election must be the winner. And every other non-crashed process must agree on this.

The bully election algorithm

Additional assumptions

  • Reliable links
  • Possibility to decide who has crashed (synchronous system)

Algorithm

  • When any process P notices that the coordinator is no longer responding to requests, it initiates an election by sending an ELECT message, including its ID, to all processes with higher IDs.
    • If no-one responds (meaning they are not up), P wins and sends a COORD message to the processes with lower IDs.
    • If a process P' receives an ELECT message, it responds (stopping the former candidate), and starting a new election.
  • If a process that was previously down comes back up, it holds an election and if it's the higher ID process currently running, it takes over the coordinator's job.

A ring-based algorithm

Processes are logically arranged in a ring, ordered by their process id.

When a process detects a leader failure, it sends an ELECT message containing its ID to the closest alive neighbor.

  • Upon a receipt of the election message, a process P:

    • If P is not in the message, add P and propagate to next alive neighbor.
    • If P is in the list, change message type to COORD, and re-circulate

  • On receiving a COORD message: a node considers the process with the highest ID as the new leader.

Collecting global state

Global state of a distributed system: consists of the local state of each process, together with the message in transit over the links.

Notice that capturing the global state of a distributed system would be easy if we could access a global clock, but we do not have one.

Cuts & distributed snapshots

A distributed snapshot reflects a (consistent, global) state in which the distributed system might have been, but we cannot be exactly sure since we are not able to capture the state of every node and every channel in the exact same time.

When reconstructing the global state, we need to take care to preserve consistency. In fact, if a message receipt is recorded, the message sending must as well, but the contrary (a sent message doesn't have to be receipt) is not required.

Cut: the union of the histories of all processes up to a certain event
A cut is consistent if for any event e it includes, it also includes all the events that happened before e.

Distributed snapshot protocol - Chandy Lamport protocol

Guarantees
Distributed snapshot by Chandy-Lamport is a protocol for taking pictures of the global state:

  • without interrupting the distributed system
  • selecting consistent cut.

Assumption

  • FIFO
  • Reliable links and nodes
  • Strongly connected graph

Description

  • Any process p may initiate a snapshot by:
    1.1 Recording its internal state
    1.2. Sending a token on all outgoing channels --> this signals a snapshot is being run
    1.3. Start recording a local snapshot: record messages arriving on every incoming channel.
    That is, if a message arrives on a channel being recorded, record the arrival of the message, then process the message as normal.
  • Upon receiving a token, a process q:
    • If not already recording local snappshot:
      2.1 Stop recording incoming messages on the channel the token arrived along
      2.2 execute steps 1.1, 1.2, 1.3
  • Termination:
    - Each process considers the snapshot ended when tokens have arrived on all its incoming channels.
    - Afterwards, the collected data can be sent to a single collector of the global state.

Observations
If a snapshot is started at more than one location at the same time, each token is associated to each snapshot, in order to be able to distinguish.

Several variation of the protocol, regarding the protocol itself (es: including only nodes which sent/received messages, called incremental snapshot), or the collection of result of the snapshot.

Termination detection

The goal of termination detection is to know when a computation has completed or deadlocked, that is:

  • All processes should be idle
  • There should be no messages in the system

Simple approach

For doing this, a distributed snapshot can be used, under the condition that channels are empty when the snapshot is finished. Not detailed here.

Dijkstra-Scholten termination detection

Based on the fact that we are in diffusion computation, that is:

  • initially all process are idle except the init process.
  • processes are activated only by a message being sent to it
  • The termination condition is the one described above.

Description

  1. Create a tree out of the active processes
  2. When a node finishes processing and it is a leaf, it can be pruned from the tree
  3. When only the root remains, and it has completed processing, the system has terminated.

Tree creation and leaf identification

  1. Each node keeps track of the nodes it sends a message to (who may be waken up).
    • If a node is waken up, it is added to the three as child of the sender
    • If a node was already awake when the message arrived, then it is already part of the tree, and should not be added as a child of the sender
  2. When a node has no more children and it is idle, it tells its parent to remove it as a child

Comparison

Overead:

  • Simple approach: one message per link plus cost to collect result.
  • Dijkstra-Scholten approach: depends on the number of messages in the system, but do not involves never-activated processes.

Termination

  • Simple approach: if the system is not terminated, it need to run again.
  • Dijkstra-Scholten: termination detected when the last ack is received.

Distributed transactions

Transactions can be:

  • Flat: ACID transactions, which are all or nothing transactions
  • Nested: are constructed from sub-transactions, which can be undone once committed in case their parent transaction aborts. Sub-transactions conceptually operate on a private copy of the data:
    • If it aborts, the private copy disappear
    • If it commits, the modified private copy is made available to the next sub-transaction.
      Typically, each sub-transaction runs on a different host, providing a given service.
  • Distributed: essentially flat transactions on a distributed data. There is a need for a distributed locking.

Achieving atomicity

There are two main approaches to achieve atomicity in a transaction:

  • Approach 1: private workspace
  • Approach 2: writeahead log

Private workspace

The private workspace approach is a pessimistic method, assuming that operations may fail and thus preparing for potential rollbacks.

In this approach, modifications made by a transaction are initially applied to a separate memory space, creating shadow copies or shadow blocks of the original data. This private workspace isolates changes from the main data, allowing the transaction to proceed without immediately affecting the original dataset.

If the transaction is aborted, this private workspace is simply discarded, leaving the main data untouched. If the transaction completes successfully, the private workspace is merged into the main data, making the changes permanent.

Itâ€™s worth noting that the private copy is not a full duplicate of the parentâ€™s data. Instead, it consists of copies of indexes or references to the original data blocks, rather than duplicating the entire data structure. This means that the data itself remains in a central memory space, with both the main data and the private workspace pointing to the same data blocks. As the transaction proceeds, modifications are applied to this workspace, allowing new data blocks to be added or existing blocks to be modified.

Writeahead log

The private workspace approach is a optimistic method.

Files are modified in pace, but a log is kept reporting each operation with:

  • Transaction that made the change
  • Which file/block has been changed
  • Old a new values

If the transaction succeeds, the commit is written to the log, and the file is actually modified only after the file is actually modified.

If the transaction aborts, the original state restored based on logs, starting by the end (rollback)

Distributed scheduler

General organization for distributed transactions

The general organization of managers for handling distributed transactions is:
- Transaction manager: is the component that transforms high-level operations in scheduling requests. Typically there is one for distributed transactions, acting as a coordinator.

  • Scheduler: the component that guarantees consistency and isolation, by determining which transaction can pass an operation to the data manager and when. Can be distributed over several hosts.
  • Data manager: the component that actually execute read/writes on data

Serializability

Of the various intervealings of operations possible, only the ones corresponding to some linearization of the involving transactions are legal.
Implementable with:

  • Mutual exclusion (locks)
  • Explicit operation ordering:
    • optimistic
    • pessimistic

Two-Phase Locking (2PL)

Two-Phase Locking (2PL) is a protocol for managing concurrent transactions to ensure serializability:

Lock Acquisition: When a transaction needs data access, it requests a lock from the scheduler. If a conflict arises with an existing lock, the operation is delayed until the lock is released.

Phases:

  • Growing Phase: A transaction can acquire locks but cannot release any.
  • Shrinking Phase: Once it starts releasing locks, it cannot acquire new ones.

Strict 2PL: Requires all locks to be held until transaction termination, preventing cascading aborts.

Pros and Cons:

  • Serializability is guaranteed, but deadlocks may occur.

Widely used for its reliability in maintaining data consistency.

2PL can be implemented as follow:

  • Centralized 2PL: Thereâ€™s one main "lock manager" that controls access to data. Transaction manager, executing a transaction, ask this manager for permission (a lock) before accessing data. This central manager keeps things organized but can become a single point of slowdown.
  • Primary 2PL: here are multiple lock managers, each in charge of specific data item. For each of them, thereâ€™s a "primary" copy on one server, and the lock manager on that server controls access to it. Transactions go to the right lock manager based on where the primary copy of the data is.
  • Distributed 2PL: Data is spread (replicated) across many servers, with each server lock manager managing locks for its own data copies. Transaction manager, executing a transaction, ask the server where copy of the data is located for a lock, allowing for more flexibility, but itâ€™s harder to keep eacheverything in sync.

Timestamp ordering

This approach is based on timestamps, which can be assign for example by using Logical clocks.

Pessimistic timestamp ordering

Each data item x has a:

  • Write timestamp: referring to the timestamp of the last transaction which wrote it
  • Read timestamp: referring to the timestamp of the last transaction which read it

Description
The scheduler operates as follow:

  • When receives write(T,x) at time=ts
    • If ts> and ts> perform tentative write with timestamp
    • else abort T since the write request arrived too late
  • Scheduler receives read(T,x) at time=ts
    • If ts>
      • Let be the latest version of x with the write timestamp lower than ts
      • If is committed perform read on and set =max(ts, )
      • else wait until the transaction that wrote version commits or abort then reapply the rule
      • else abort T since the read request arrived too late

Note: aborted transactions will reapply for a new timestamp and simply retry.
This approach is deadlock-free.

Optimistic timestamp ordering

Based on the assumption that conflicts are rare, so the rationale is: do what you want without caring for others, fix conflict later.

Description

  1. Each transaction gets a start timestamp when it begins.
  2. Each data item has a last-modified timestamp showing when it was last changed.
  3. The transaction runs without locking data, assuming no conflicts.
  4. Commit Check: At commit time, the transaction checks if any data it accessed has been modified by another transaction (i.e., has a newer timestamp).
    • If no data was changed, it commits.
    • If any data was changed, it aborts and may retry.

It is deadlock free and allows maximum parallelism, however under heavy load, there may be too many rollbacks. Thus is not widely used, especially in distributed systems

Distributed deadlocks

Approaches to distributed deadlocks:

  • Ignore the problem: most often employed
  • Detection and recovery: typically by killing one of the processes
  • Prevention
  • Avoidance: never used in distributed systems, as it implies a priori knowledge about resource usage.

Centralized deadlock detection

Each machine maintains a resource graph for its own resources and reports it to a coordinator which can build a global wait-for-graph, which can be done in different ways:

  • By updating the coordinator each time an arc is added or deleted
  • By sending periodically updates about arcs
  • Coordinator can request information on demand

None of these approaches work well because of false deadlocks.

Distributed deadlock detection

Processes are allowed to request multiple resources simultaneously, let deadlock occur, then checking for deadlocks in the following way:
when a process gets blocked, it sends a probe message to the owner of the resources it want to access. The message is composed by (initiator, sender, receiver). A cycle is detected if the probe makes it back to the initiator.

Recovering from a deadlock

When a cycle is found one of the two it's possible:

  • the initiator commits suicide
  • the initiator kills the process with higher id involved in the cycle.

In practice, 90% of all deadlock cycles involve just 2 processes, at least in databases.

Deadlock prevention

Requires making deadlocks impossible by design, for example using global timestamps:

  • wait-die algorithm: If a process is getting blocked for accessing a resource, let the process wait only if its timestamp is smaller (it is older) than the one of the process which is holding the resource. That is, processes can wait only if they are older than the one holding the resource.
  • wound-wait algorithm: if a process can be preempted (its resource can be taken away from it), then preempt the young process by making it abort the transaction. The process may immediately try to reacquire the resource, but it will wait.
    Notice that in wait-die, young processes can die several times before the old one releases the resource, in the wound-wait it doesn't happen.
