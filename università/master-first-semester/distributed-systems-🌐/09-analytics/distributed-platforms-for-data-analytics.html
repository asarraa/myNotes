Distributed Platforms for Data Analytics

From a technical perspective, collecting the data for data science purposes is very different with respect to traditional information systems. In fact, in the latter decisions about what data to keep and why is made upfront, whereas in data science it is better to keep more data as possible, since that deciding what to keep is more expensive than to keep it all.

Consequences
Keeping all data instead of filtering or selecting specific data introduces several key challenges:

  1. Volume (Data at Rest): Storing everything means dealing with massive amounts of data, which requires significant storage capacity and efficient management of large datasets.

  2. Velocity (Data in Motion): Data will arrive at high speeds, and the system must be able to ingest, process, and analyze this data in real-time or near real-time without bottlenecks. This because information loses value as it becomes old in many domains.

  3. Variety (Many Formats): The data will come in different formats (structured, unstructured, semi-structured) from various sources, making it necessary to handle multiple versions and formats simultaneously.

  4. Veracity (Uncertain Accuracy): Not all the data will be accurate or reliable, so the system must incorporate mechanisms to detect and address errors or inconsistencies to ensure the quality of analysis.

Problem scope

Beyond scalability to large data volumes and support for quickly changing data, other required functions are:

  • Automatic parallelization & distribution
  • Fault-tolerance
  • Status and monitoring tools
  • A clean abstraction for programmers

Scalability

When facing a problem, many difficulties born when it is considered at scale. This is especially true for data intensive applications. In many of these cases:

  • Reading input and reading or storing the intermediate state of the computation can become the bottleneck
  • A distributed solution is needed if input/state do not fit in one machine, so also coordination and communication can become the bottleneck
    Thus, it is important to look at the scale of the problem before doing assumptions.

Platforms

MapReduce

MapReduce was developed by Google in 2004, based on two phases: mapping and reducing, both highly parallelizable. Easy to use since manage the complexities automatically.

Beyond MapReduce

In the last decade, many systems extended and improved the MapReduce abstraction in many ways:

  • Dataflow model: unlike MapReduce which relies on just two processing steps, many modern systems support more phases, allowing the creation of direct acyclic graph of transformations.
  • Iterative computations: Newer systems support looping and iterative computations more efficiently by allowing data to remain in memory and be re-used across iterations, improving performance for tasks like PageRank or neural network training.
  • Stream processing support: While MapReduce is mainly designed for batch processing (processing large, static datasets), modern systems also support stream processing which deals with continuous data flows in real-time, enabling low-latency processing as new data arrives (e.g., processing live sensor data or real-time analytics).
  • Memory handling: Modern systems increasingly use in memory processing, storing and processing data directly in RAM to significantly reduce latency and increase speed. Some systems even adopt hybrid approaches, combining disk and memory.

The following sections show two systems as representatives of two different architectural approaches: Task Scheduling and Task Pipelining.

Important

The different problems one can have are orthogonal to the type of architecture adopted (scheduling vs pipelining), those are about the nature of data input, in particular they are:

  • Batched processing: computation about data that don't change in time
  • Streaming processing: data continuously change and you want to continuosly update the solution to the problem as your data arrive

However, each of the two architectures are better than the other for one of these two cases.

Apache spark - Scheduling architecture

Apache Spark extends MapReduce with an arbitrary number of stages. It offers in-memory caching for intermediate results, which optimizes performance when data is reused across multiple stages.

In Spark's architecture, the input data is divided into micro-batches, which are processed in parallel. These batches are scheduled across multiple nodes for different phases of computation, making Spark particularly effective for batch data processing.

Apache Flink - pipelining architecture

Apache Flink uses a pipelined execution model, where jobs are not split into discrete stages. Instead, operators are instantiated as soon as the job is submitted, and data flows continuously between them through TCP channels. As soon as an operator receives data, it starts processing, enabling multiple operators to run concurrently.

Flink excels in stream processing applications due to its low-latency pipeline architecture. Data is processed as it arrives, rather than waiting for task scheduling, making it highly efficient for real-time analytics. However, the same pipelined approach can be applied to batch processing as well, where the entire batch is treated as a data stream.

Comparison

Latency: It is how long does it take for the output to be updated after you receive the input. Pipeline approach of Flink provides lower latency. In fact, new data elements are ingested into the network of operations as soon they become available without the needing to accumulate (micro-) batches. This is a very relevant property for streaming processing.
Throughput: Scheduling approach of Spark offers more opportunities to optimize throughput, in fact moving larger data blocks can be more efficient and scheduling decisions can consider data distribution.
Load balancing: Scheduling approach of Spark simplifies load balancing since dynamic scheduling decisions can consider data distribution.
Elasticity: it is the possibility of a system to dynamically adapt resource usage to the load of the system. Scheduling approaches are better for elasticity as scheduling decisions can dynamically change the set of physical resources being used.
Fault tolerance: in scheduling approaches if a node fails the task is re-scheduled while pipelining approaches rely on checkpointing, precisely with snapshot using Chandy Lamport algorithm and replay from the last available one. For example Spark relies on lineage (no intermediate results are stored, if there are crashes and data are lost, simply recompute them).
