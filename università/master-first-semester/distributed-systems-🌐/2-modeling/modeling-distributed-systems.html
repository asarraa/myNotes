The software architecture of a distributed system

Q: what are the two major types of software architecture of a distributed systems? Describe them
A: The following

  • Network OS based: The communication services are provided by the network OS. Since different machines can have different network OSes, it's up to the programmer to mask those differences
  • Middleware based: middleware provides advanced communication services, it masks most of the platform differences.

The run-time architecture

Q: What is a run-time architecture of a distributed system?
A: Refers to the dynamic structure of the system when it is executing identifies the class of components that build the system, the various types of connectors and the data types exchanged at run-time.

Q: what is an architectural style ?
A: An architectural style refers to a set of design patterns and principles that define how components in a system interact. It represents the high-level design or blueprint of the system, dictating how components are organized and how communication occurs between them.

Client - server architecture

Q: What is the most common architectural style today? Describe it
A: Client-server. Servers provide a set of services through APIs, users access those services through client, which invoke APIs with a message based communication (RPC).
Services can be divides in three classes:

  • User interface services
  • Application services
  • Storage services
    Multi-tiered client-server applications can be classified looking at the way such services are assigned to the different tiers.

Service oriented architecture - SOA

Q: What's the concept of Service Oriented Architecture (SOA)?
A: Is a specific version of the client-server architecture where Services, which are units of functionality, are exported by service providers. Brokers hold the description of available services to be searched by interested consumers, which bid and invoke the service they need.

Q: In SOAs, what's orchestration?
A: In the service oriented architecture, the orchestration is the process of invoking a set of services in an ad-hoc workflow to satisfy a given goal.

Q: In Web Services (special type of SOA), what are UDDI, WSDL, SOAP?
A: Protocols and languages used in the SOA architecture:

  • UDDI: the protocol we use to interact with the broker.
  • WSDL: the language in which service contract is written.
  • SOAP: protocol used by the service consumer to invoke the service. It is build on the HTTP protocol and has parameters in XML.

Representational State Transfer style

Q: What is the REST architectural style?
A: The REST (REpresentational State Transfer) architectural style is a set of principles defining how Web standards should be used. It is based on the idea that interactions must be stateless, where each client request is independent of the server's state.

Q: How does REST differ from SOA in terms of state management?
A: In REST, interactions are stateless, meaning the state is kept by the client and sent with each request to the server. In contrast, SOA (Service-Oriented Architecture) allows for stateful interactions, where the server can maintain part of the interaction's state (e.g., an online shopping cart).

Q: What are the main pros and cons of the REST architecture?
A:
Pros:

  1. Scalability: No need to share state between multiple servers, making the architecture easily scalable.
  2. Caching: REST makes caching easier, which helps improve scalability by reducing the load on the server.
    Cons:
  3. The programming of the client is made more complex because the client must maintain and manage the state, sending it with each interaction.

Q: What are the key goals of REST?
A:

  1. Scalability
  2. Generality of interfaces
  3. Independent deployment of components
  4. Use of intermediate components to reduce latency, enforce security, and encapsulate legacy systems.

Q: What are the main constraints of REST?
A: The following

  • Client-Server: All interactions are client-server.
  • Stateless: Each interaction is stateless, meaning the client must send the state with each request.
  • Cacheable: Data in responses must be labeled as cacheable or non-cacheable.
  • Layered System: Components can only interact with the immediate layer they are communicating with.
  • Optional: Clients may support code-on-demand.
  • Uniform Interface: Components expose a uniform interface. The most of the information about what you want the service to do is done by encoding parameters.

Q: What are the four constraints of the uniform interface in REST?
A: 1. Identification of resources
2. Manipulation of resources through representations
3. Self-descriptive messages
4. Hypermedia as the engine of application state

Q: What is the Identification of Resources constraint in REST?
A: In REST (Representational State Transfer) architecture, every resource—such as data or services—must have a unique identifier, usually represented as a URL (Uniform Resource Locator). This means that any entity that can be identified, like a user, a document, or a service, can be accessed or manipulated using its unique URL. This helps maintain clarity and organization in web communication, allowing clients to interact with specific resources easily.

Q: How are resources manipulated in REST?
A: REST components manipulate resources by transferring a representation of the resource (e.g., XML or JSON) rather than the resource itself. A representation in REST consists of data and metadata that describes the data

Q: What is meant by Self-descriptive messages in REST?
A: In REST, self-descriptive messages contain control data that defines the purpose of the message, such as the action being requested (e.g., GET, POST) or the meaning of a response. It can also parameterize requests and override default behaviors, such as cache settings.

Q: What is Hypermedia as the engine of application state in REST?

A: In REST, clients move between different states by processing resource representations, usually linked to other resources via hypermedia links. This allows the application state to evolve dynamically as clients navigate through the links.
When interacting with an online book store for example, the client’s state evolves by following hypermedia links provided in each response, allowing it to progress from browsing books to checking out without hardcoding the navigation structure. Each step is dynamically guided by the server’s links as follows:
Initial Request: The client starts by sending a GET request to /books. The server responds with a list of books and links to each book's details.
{ "books": [ { "title": "Book A", "link": "/books/1" }, { "title": "Book B", "link": "/books/2" } ] }
Navigating Links: The client follows the link to /books/1 to get more details about "Book A". The response includes a link to "Add to Cart."

Peer-to-Peer

Q: Describe the Peer-to-Peer architecture and what is good about it
A: In the Peer-to-peer architecture all component play the same role, there's no distinction between client and server. It avoid centralization, which is good for not having a single point of failure.

Object-Oriented architecture

Q: What is an object-oriented distributed system?
A: An object-oriented distributed system applies object-oriented programming (OOP) principles across multiple machines. Distributed components encapsulate a data structure providing an API to access and modify it, similarly to objects in OOP. Component interact through RPC.

Q: What are tiers in an object-oriented distributed system?
A: Tiers in an object-oriented distributed system represent different layers of components, each encapsulating its own state and exposing methods, similar to objects in traditional OOP. These tiers are distributed across machines but behave like local objects.

Q: How are objects in object-oriented architecture described?
A: Objects are described using a descripting language which is indipendent by the implementatio language

Q: what are the pros of object-oriented architecture?
A: The pros of object oriented programming (information hiding, encapsulation, easy reuse of objects, wrapping of legacy objects)

Data-centered architecture

Q: What is the core idea behind a data-centered system?
A: In a data-centered system, components communicate indirectly through a shared repository. Components can put, get, or remove data from this repository, and all communication is mediated by it, with no direct interaction between components. Communication with the repository is (usually) through RPC.

Q: What operations can components perform in a Linda-based tuple space?
A: In a Linda-based tuple space, components can perform three operations:

  • out(t): Insert a tuple into the space
  • rd(p): Read a tuple (non-destructive)
  • in(p): Take a tuple (destructive)

Q: What is an advantage and what is a disadvantage of using a data-centered system with a shared repository?
A: An advantage is that components don't directly address each other, allowing for dynamic reconfigurability at runtime. This decoupling makes it easier to modify or scale parts of the system without disrupting the whole architecture.
A potential downside is that the shared repository can become a bottleneck, especially in high-concurrency environments, as all communication passes through this central point. Also, the model is only proactive, since processes explicitly request a tuple query.

Q: Why is the data centered model revitalized today?
A: For distributed computing.

Event based

Q: What's the idea behind the event based architecture? What type of communication is used to implement this architecture?
A: Components collaborate by exchanging information about events. In fact, they can publish notifications about the events they observe, or subscribe to events they are interested to be notified about.
The event-based architecture is purely message based, in most of the times it uses a publish-subscriber communication model.

Q: What's the main difference between event based and data centered architectures?
A: Unlike in data centered architecture, in the event based one the communication is necessarily multicast, as the one who public something don't know how many subscribers there are (and who are: anonymity).

Q: What is an event bus? Where is it used?
A: There's a transport layer, called event bus that allows event to reach components which are subscribed fro that specific event, similarly to the tuple space in Data centered architecture, the difference is that the communication is necessarily multicast.

Q: Is the event based model proactive or reactive? Why?
A: Proactive, you don't search for event but event search you.

Mobile code technologies

Q: What is the mobile code architecture?
A: Mobile code architecture leverages the ability to move code between hosts at runtime. It allows for different paradigms where code execution can occur remotely or on the client side.

Q: What's strong mobility and what's weak mobility?
A: the following

  • Strong mobility is the ability of a system to allow migration of both the code and the execution state of an executing unit to a different computational environment.
  • Weak mobility is the ability of a system to allow code movement across different computational environments.

Q: How does Remote Evaluation work in mobile code paradigms? A: In Remote Evaluation, the client sends code to the server for execution. The client requires specific functionality but lacks the resources to execute it. An example is AWS Lambda, where users upload code to be executed on the server.

Q: What is the difference between Code on Demand and Remote Evaluation?
A: In Code on Demand, the client requests specific code from the server, which then sends the code for local execution. In contrast, Remote Evaluation involves the client sending code to the server for execution. A common example of Code on Demand is JavaScript in web browsers.

Q: What are Mobile Agents in mobile code paradigms? A: Mobile Agents are processes that carry their code, state, and execution context as they move between hosts. This approach can pose security risks, as the mobile agent could potentially be harmful, making it less commonly used than other paradigms.

A: Security: securing mobile code applications is a mess

The interaction model

Q: What is a distributed algorithm and which factors influence his behavior?
A: A distributed algorithm is a definition of steps taken by each process, including the transmission of messages between them. His behavior is influenced by:

  • The rate at which each process proceeds
  • The performance of the communication channels
  • The different clock drift rates

Q: Describe synchronous and asynchronous distributed systems and what is a property that relate the two?
A: follows

  • Synchronous distributed systems: there are lower and upper bounds on time of execution of process steps, message exchanging time, clock drift rate.
  • Asynchronous distributed systems: there are no such bounds. Usually distributed systems are asynchronous.
    Property: any solution that is valid for an asynchronous distributed system is also valid for a synchronous one.

Q: Explain what's the Pepperland example and what conclusion can we take from that.
A: he Pepperland example illustrates the Byzantine Generals Problem, which highlights the challenges of achieving distributed agreement in asynchronous distributed systems. In this scenario, two divisions of the Pepperland Army (the Apple and Orange divisions) are camped atop hills, facing an enemy, the Blue Meanies, positioned in between.
For a successful attack, both divisions must attack simultaneously; if one division attacks alone, the attacker will be defeated. Conversely, if both divisions stay in camp (0 attack), they remain safe. The key challenge is for both divisions to agree on a common decision about who will lead the attack and when to charge.
In an asynchronous distributed system, it becomes difficult for the divisions to reach a safe agreement. They can attempt to communicate by sending messages to each other (e.g., reporting the number of soldiers left), and the division with the most soldiers would take the lead, with a predefined tiebreaker if needed. However, due to potential message delays, there's uncertainty about when to safely charge, making coordinated action risky.
In a synchronous distributed system, the divisions can agree to charge after a maximum delivery time, ensuring that all messages are received in a timely manner. This guarantees that both divisions can safely coordinate their actions.
The conclusion from the Pepperland example emphasizes that achieving consensus in distributed systems is significantly easier in synchronous environments, where message delivery times can be guaranteed, compared to asynchronous environments where delays can lead to uncertainty and potential failure in decision-making.

Failure model

Q: What does a failure model define?
A: a failure model defines the ways in which failure may occur to provide a better understanding of the effects of failures.

Q: Is it possible to reach a consensus in asynchronous distributed systems in presence of one faulty process? Why?
A: Fischer, Patterson demonstrated that in an asynchronous systems there is no way of agreement, in the presence of even one faulty process. In practice we solve it by changing the assumptions
