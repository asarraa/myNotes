FLEX - Scanner generator

Fast lexical analyzer

Input: a specification file of the scanner
Output: a C source code file that implements the scanner

File format

Divided in:

  • Definitions: associate a name to a set of characters
  • Rules: Token to be recognized and the semantic action to be performed at each match. Semantic actions may use: yytext (the matched text), yyleng (the length of the matched text).
    N.B. the semantic action can contain directly the business logic, however for complex application (compilers), they just return the token type (a lexical value associated to it).
  • User code: is c code which will then be copied into the generated scanner. Other code can be insert through the file by using the %{ [insertcodehere] %} notation.

Generated scanner

Flex generates a scanner, which is a c source code file called lex.yy.c. Inside it there is

  • FILE *yyin = stdin which is where the input comes from.
  • int yylex(void) which is the function that implements the scanner itself (c implementation of a FSA), and reads from the yyin file.

How yylex() Works:
The scanner reads tokens from the input stream (yyin) one by one.
For each token matched:
- It executes the semantic action associated with that pattern in the Flex rules file.
- If the action contains a return statement, yylex() returns that value to the caller.
- The process continues until:
1. A semantic action returns a value.
2. The end of the input file is reached (in which case yylex() returns 0).

The code written in the "user code" part of the flex file is also insert in lex.yy.c. So it is useful to have a main in that part, which will call the yylex() function.

Bison - Parser generator

Bison is the GNU Parser Generator. Parsers perform syntactic analysis.

Linking

Each source file is compiled (and assembled) separately into an object file (not executable binary).
Then linking: combining multiple object files (es: main + library) into an executable. For not have to repeat function signatures in each file that use it (library functions in the main file), we use headers, .h files which contain signatures of functions. If included in a .c file, headers allow to use the defined functions in different files. In the linking phase the different files are put together into the executable file.

Unions

Unions are like structs, but assigning a value to one item of the union invalidates the others. Since union's member overlap in memory, the size of union is the size of the biggest type size contained.

Introduction

Given a grammar that defines the syntax of a language, we need to perform syntactic analysis:

  • identify grammar structures
  • verify syntactic correctness (report conflict or solve them)
  • build a derivation tree for the input
Tokens and non-terminals symbols
  • Terminal symbols. Usually those terminal symbols are the tokens generated by the lexer. Thus, those are the inputs of the parser.
  • Non-terminal symbols are only generated through reduction of grammar rules.

Bison interface

Bison generates:

  • <filename>.tab.c: the source code of the parser
  • <filename>.tab.h: the header containing the signatures of the functions.

How do the parser work?

The main parsing function in the generated file is `int yyparse(void)

  1. Takes token from the scanner: (using the scanner yylex() function that flex-generated scanners provide)
  2. Decodes the grammar derivation: at each decoded derivation, its semantic action is invoked. Those semantic actions produce the output of the parsing process.
Important

Parser executable = scanner source (generated by FLEX) + parser source (generated by Bison)

File format

  • Prologue: headers inclusions, variable declarations can be inserted here
  • Definitions: definition of tokens, operator precedence, non-terminal types
  • Rules: grammar rules and related semantic actions
  • User code: C code (helper functions)

Definitions

Tokens definitions are "exported" to the scanner by means of an enum inserted in the .tab.h header.

Rules

The grammar in BNF notation is inserted here, a semantic action is inserted at the end of each rule alternative:

sections : sections section
			{/* semantic action to perform after the rule is               reducted*/}
		| /*empty string*/
			{/* semantic action*/}
		;
section: LSQUARE ID RSQUARE options
			{/* semantic action*/}
		;
options: options option
			{/* semantic action*/}
		| option
			{/* semantic action*/}
		;
option : ID EQUALS NUMBER
			{
			/*semantic action*/
			}
		| ID EQUALS STRING
			{
			/* same */
			}
	;

Semantic actions are executed when the rule they are associated with has been completely recognized, thus the order of execution is bottom-up w.r.t. the syntactic tree.

Semantic values

To track what each token/non-terminal represents, we associate a variable to each. How:

  • tokens (terminals): the value is associated in the lexer
  • non-terminals: the value is assigned in the semantic action of that non-terminal. That is, as soon as a reduction occurs, the semantic action to the occurred alternative assigns the semantic value of the non-terminal that has been reduced.
Declaration of semantic values

The types of each semantic value are specified in the definition section:

  • For tokens, the type specification is in the token declaration.
  • For non-terminals, the type specification is in a special %type declaration.

Convention: token names are in capital letter, non-terminal are in non-capital letter.

Accessing semantic values in actions

The semantic value of each grammar symbol in a production is a variable called $i, where i is the position of the symbol. $$ corresponds to the rule itself, which is used to assign the value of a non-terminal.

Setting semantic values in the scanner

Bison declares in the parser a global variable yyval which is of type union. This union contains every data type used in the parser.
As soon as the scanner recognizes a token, in its semantic action before returning the token, sets the recognized value in the yyval. Notice that using a union the scanner can assign variables of different types to the same global variable.

Example flex semantic action
[0-9]+			{
					yylval.value = atoi(yytext); //sets token value
					return NUMBER; //return token
				}

Precedence and associativity

Bison allows to handle precedence in a simpler way with respect to grammars which must encode precedence using left-recursive and right-recursive rules.

Precedence

In the Definitions part of the file we list the tokens in order of growing precedence with the %precedence declaration. That is, lowest precedence at the top, highest precedence at the bottom. Tokens on the same line have the same precedence.

  • Rules take precedence from the last token

Associativity

To specify associativity, replace %precedence with one of the following:

  • %left
  • %nonassoc
  • %right

Those keywords keep the meaning of precedence but also specify associativity.

Compilers

A compiler translates a program written in a language into a semantically equivalent program expressed in language

Pipeline

It's organized as a pipeline:

  1. Front-end: from source language to intermediate form. It involves:
    • the lexical analysis (performed by scanner)
    • syntactic analysis (by parser)
    • semantic analysis
  2. Middle-end: transformations and optimizations on the intermediate form
  3. Back-end: from intermediate form to target machine language

LLVM

One of the most common frameworks to build compilers is LLVM. It includes a suite of tools that includes a set of libraries for writing compilers.
It allows you to translate source code into an intermediate representation (IR) that is independent of the language and platform.
The front-end provided by LLVM for c-like languages is Clang.

Clang and GCC

  • Clang:
    • It is the front-end compiler for the LLVM project.
    • It produces intermediate code (LLVM IR), which can be optimized and converted into machine code for various architectures.
    • Designed to be modular and easily extensible.
  • GCC:
    • Part of the GNU suite, it is an "all-in-one" compiler that generates machine code directly.
    • Less modular compared to LLVM/Clang but very mature and stable.

ACSE

ACSE is a simple compiler:

  1. accepts a C-like source language (called LANCE)
  2. emits RISC-V (RV32IM) assembly language

Helper tools:

  • asrv32im: Assembler (from assembly to machine code)
  • simrv32im: RISC-V simulator

Otherwise there is RARS, which is a RISC-V assembler and runtime simulator.

Root rules

A LANCE source file is split into two sections:

  1. Variable declarations: var_declarations
  2. List of statements: statements
  • IDENTIFIER is the non-terminal for a generic name
  • statement: unit of programming that express some action to be carried out:
    • simple: assignment, read, write ...
    • compound: if, while, do-while
  • exp: generic expression
    • NUMBER
    • var_id
    • operations
  • var_id: generic variable name

Other important stuff:

  • Precedences
  • Weird unary minus operator

Inside ACSE

Core elements of ACSE compiler:

  • Scanner: flex source in scanner.l
  • Parser: bison source in parser.y
  • Codegen: signatures of functions that generate assembly instructions in codegen.h

What happens during compilation?

  • Nodes in the syntax tree are associated to some C code
  • Tree is visited bottom-up, left-to-right and the C code is executed, this means:
    • RISC-V assembly code takes the place of the node
    • Optional semantic value is accessible to the parent of the node
  • The assembly code gets written in textual form to the output file (and doesn't get executed at compile time, but at runtime, that is when the compiled program is executed)

The intermediate representation (IR)

IR is a data structure used in compiler to represent the program. This is where the assembly code is stored during parsing. In ACSE, composed by:

  • Instruction list
  • Symbol table

As some LANCE instructions are recognized and the relative semantic actions are executed, the instruction list and the symbol table are modified to store RISC-V assembly code (instructions and symbols).

Info

Actually, the instructions saved in the IR instruction list are not the actual assembly code but can be different, since we still are in the front-end phase there are more degrees of liberty (es: infinite registers can be used), only in the back-end phase the translation to machine language (assembly or binary) will be performed. In the case of ACSE, the IR contains a superset of the RISC-V assembly code.

The program data structure

At the top of the parser.y file the program global structure is declared. It contains the IR and other contextual information.

Since the assembly code needs to be stored in the IR, almost every C function in the semantic actions takes program as parameter.

Registers identifiers

A register identifier represents a given register in the bank of infinite registers. Since they are only temporary registers that will be mapped to actual register later, they are indicated with temp<n> where n is the number of the register.

All the generation functions' signatures are in codegen.h, while the function used by those (es: getNewRegister) are in program.h

Nested operators

Gli operatori che possono essere annidati possono creare problemi poiché in alcuni non basta inserire i valori semantici "utili" all'interno del token utilizzato.
Si prenda come esempio il while_statement:

while_statement
: WHILE
{
// Assign a label at the beginning of the loop for the back-edge.
$1.lLoop = createLabel(program);
assignLabel(program, $1.lLoop);
}
LPAR exp RPAR
{
// Generate a jump out of the loop if the condition is equal to zero.
$1.lExit = createLabel(program);
genBEQ(program, $4, REG_0, $1.lExit);
}
code_block
{
// Generate a jump back to the beginning of the loop after its body.
genJ(program, $1.lLoop);
// Assign the label to the end of the loop.
assignLabel(program, $1.lExit);
}

Si noti che il valore semantico associato al token WHILE è una struttura dati contenente due campi: label corrispondente all'inizio del loop e label corrispondente alla fine del loop. Durante il parsing dello statement, tali campi vengono acceduti tramite $1 per poter gestire i branch e le jump nei vari casi previsti dal while.

Ora, le riduzioni avvengono dal basso verso l'alto, ciò vuol dire che il primo while che sarà ridotto è quello più interno. Per ogni token di cui tale statement è composto, verranno eseguite le azioni semantiche sopra descritte, non ci sono problemi visto che si può accedere ai campi contenuti nel valore semantico del token WHILE. Si arriverà più tardi a ridurre il while_statement più alto: anche qui, è possibile accedere al valore semantico del token WHILE associato, perciò il nesting non dà alcun problema.
