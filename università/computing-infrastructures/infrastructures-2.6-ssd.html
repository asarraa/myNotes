SSD

Overview

SSDs are Solid-state storage devices. They are different from HDDs because they have no mechanical or moving parts, in fact they are build out of transistors, just like memory and processors. However, unlike RAM, they are capable to retain information despite power loss.

An SSD includes:

  • Solid-states memory components: These are the actual NAND flash memory chips where data is stored. There can be one or many, depending on the capacity and design.
  • Controller: It's a small embedded processor that manages how data is stored, read, written, and erased on the flash memory. Often this component becomes the bottleneck to the transfer rate

SSDs uses traditional HDD interfaces and form factor (even if this is becoming less true with tech evolution), but guarantee higher performances than HDDs.

Storing bits

SSDs use flash memory, where data is stored by trapping electrons in a floating gate transistor. The amount of charge in a cell shifts its threshold voltage (Vₜ), which is used to determine the stored value.

SSDs store bits in memory cells using different levels:

  • SLC (Single-Level Cell) stores 1 bit per cell (2 states): the cell only needs to distinguish between two voltage levels (charged or not), storing 1 bit (0 or 1).
  • MLC (Multi-Level Cell) stores 2 bits per cell (4 states): the cell must distinguish between 4 voltage levels
  • TLC (Triple-Level Cell) stores 3 bits per cell (8 states): the cell must distinguish between 8 voltage levels
  • QLC, PLC go even further with more bits per cell.

This is done by assigning reference points to divide the voltage range into multiple regions, each representing a specific bit pattern (e.g. 00, 01, etc.).

This organization implies there is a trade-off:

  • More bits per cell = more dense and cost-efficient (smaller size, cheaper).
  • But: voltage regions get tighter, making the cell more sensitive to noise, slower to program/erase, and more error-prone.

Internal organization

A NAND flash memory chip is organized in:

  • Pages: a page contains multiple logical block addresses (LBAs)--> 512B-4KB
  • Blocks: typically consists of multiple (e.g. 64) pages --> 128-256KB
Terminology

Block/Page terminology in the SSD context can clash with previous use

Memory operations

his internal organisation has also impacts on how memory operations are performed, in fact:

  • Erasing: the smallest unit that can be erased is a block (or Erase Block) --> through the ERASE command
  • Reading/writing: the smallest unit that can be read/written is a Page --> through the READ and PROGRAM command, respectively

Page states

Pages can be in three states:

  • Empty (or ERASED): do not contain data
  • Dirty (or INVALID): contain data that is no longer in use (or never used)
  • In use (or VALID): contain data that can be actually read

Writing and erasing

Three important rules apply:

  1. Only empty pages can be written
  2. Only dirty pages can be erased
  3. It is meaningful to read only pages in the VALID state

Of course, the first two rules are due to technology constraints: to erase the value in flash memory the original voltage must be reset to neutral before a new voltage can be applied

What happens when all pages are dirty and we need to write something? some page must be erased, however, we remember the smallest unit for erasing is a block, thus we can be in two cases:

  • There is a block that contains only dirty and empty pages, then this block will be erased
  • There is no block containing just dirty or empty pages exists, then special procedures should be followed to gather empty pages over the disk

Write amplification

The way in which writing and erasing happen has a consequence: in order to update a small file, the SSD might have to read an entire block, modify it, and then write the entire block back to a new location. Over time, this increases the total number of writes performed on the flash memory, causing a mismatch between the amount of information physically written and the logical data size being written. --> write amplification problem.

This problem is reflected in the performance gap that is found when testing SSDs in a controlled laboratory environment and when doing so in real application test, when SSDs are integrated in a notebook PC: while in the first case SSDs present a 15-20x improvement when compared to hard disks, in the second one performances are almost the same as HDDs.

Flash cells wear out

As we said, flash memory is built using floating-gate transistors. These transistors store data by trapping electrical charges, but the mechanism to erase them, a necessary step before rewriting, is inherently destructive over time.

In fact, each time a block of flash memory is erased, it’s hit with a large electrical charge. This process slowly damages the thin oxide layer inside the flash cell, which is critical to maintaining its charge and, therefore, the data.

Over many write-erase cycles, this degradation accumulates:

  • The silicon material itself deteriorates.

  • The electrical properties of the cells begin to change.

  • Eventually, the cell becomes unreliable, and the SSD starts to exhibit performance drops or even data retention issues.

Flash Translation Layer

Since direct mapping between logical and physical pages is not feasible, a Flash Translation Layer is used.
FTL is an SSD software/firmware component that abstracts the complexity of flash memory makes make the SSD behave like a traditional HDD to the operating system. In simpler terms, it lets the OS see a linear, block-addressable storage device, even though the underlying NAND flash works very differently.

FTL roles are about:

  1. Data allocation and address translation: FTL handles logical-to-physical address mapping: it translates the logical block addresses (LBAs) from the OS into physical flash memory locations. To reduce write amplification, the FTL must write data efficiently, ideally grouping writes together and minimizing unnecessary erases.
    A common technique used is log-structured writing, where data is written sequentially in a log-like manner: New data is always appended to the end, instead of updating it in place. In particular:
    • Each new entry (data write) is added after the previous one.
    • Old entries (outdated data) are not erased or overwritten immediately they just become obsolete.
  2. Garbage collection: TL uses garbage collection to reclaim space by erasing blocks containing invalid (dirty) pages.
  3. Wear Leveling: Since some blocks might wear out faster if they are written to more frequently, FTL performs wear leveling to that all blocks experience similar usage over time. For example, static wear leveling moves rarely changed data to different blocks to make room for dynamic data in the least used ones.

Garbage collection

The process of garbage collection is simple in case of fully garbage blocks, while it is more complex for blocks containing both valid and invalid pages (partial block). In the latter case, the steps to perform are:

  • Find a suitable partial block
  • Copy non-garbage pages
  • Reclaim (erase) the entire block for writing

Since the process of garbage collection requires reading and rewriting of live data, it is an expensive process. Of course, the cost depends on the amount of data blocks that have to be migrated. In the ideal case, only blocks that consist of only dead pages should be reclaimed.

Solutions to alleviate the problem of costly garbage collection:

  • Over provision the device by adding extra flash capacity, in order to being able to delay the cleaning.
  • Run the garbage collection in the background using less busy periods for the disk.

The ambiguity of deletes

While performing background garbage collection, the SSD assumes to know which pages are invalid. The problem is:

Most file systems don't actually delete data, but only remove meta-data

Linux delete

On Linux, the “delete” function is unlink() and it removes the
file meta-data, but not the file itself

For solving this ambiguity, a new SATA command is used: TRIM (SCSI-UNMAP)
Using this command, the OS tells the SSD that specific LBAs are invalid and may be garbage collected.

Address Translation (SSD Mapping)

Modern SSDs use mapping tables to translate logical addresses (used by the OS) to physical addresses (actual flash memory locations). However, page-level mapping can become expensive in terms of memory usage:

Example

A 1TB SSD with 4KB pages and 4-byte entries per page needs 1GB of DRAM just for the mapping table.

To reduce memory requirements and improve performance, several techniques are used:

1. Block-Based Mapping

Maps data at the block level instead of the page level.

  • Less precise, but requires a smaller mapping table.
  • Suffers from inefficiencies on small writes, since whole blocks must be updated even for small changes.

2. Hybrid Mapping

Combines block-based and page-based mapping.

  • Uses two tables:
    • Data blocks: block-mapped.
    • Log blocks: page-mapped (to handle recent or small updates).
  • Reduces write amplification and improves flexibility.
  • When data is accessed, the FTL checks both tables to find the correct location.

3. Page Mapping with Caching

Maintains full page-level mapping, but only caches the active parts of the table in memory.

  • Based on data locality: if only a subset of pages is frequently accessed, their mappings stay in cache.
  • Advantages:
    • Achieves near page-level performance with much less memory.
  • Downside:
    • Cache misses add overhead (as the mapping must then be fetched or reconstructed).

The “Small Write” Problem

In block-based or hybrid systems, small writes can be inefficient:

  • The FTL may need to copy live data from old blocks into new ones just to accommodate a small update.
  • This increases write amplification, degrading performance over time.

Wear Leveling

Blocks should wear out at roughly the same time. Even if log-structured approach and garbage collection helps in spreading writes, a block may still consist of cold data. For this reason, the FTL must periodically read all the live data out of such blocks and re-write it elsewhere. Of course, this way wear levelling increases the write amplification of the SSD and decreases performance.

This is implemented by maintaining a EW (erase/write) cycle counter for each block and acting so to maintain |max(EW cycle) - Min(EW cycle)| < e

SSD vs HDD

Unlike HDDs, SSDs are not affected by data locality and must not be defragmented, since it may damage the disk.

Let's define two metrics:

  • UBER (unrecoverable bit error ratio): a metric for the rate of occurrence of data errors, equal to the number of data errors per bits read.
  • TBW (terabytes written): it's the total amount of data that can be written into an SSD before it is likely to fail.

SSD's memory cells can accept data recording between 3,000 and
100,000 during its lifetime. Once the limit value is exceeded, the cell "forgets" any new data. Of course by leveraging wear leveling algorithms that distribute write operations across all available memory cells, even with 3,000 cycles per cell, the drive can handle millions of write operations by spreading them around.

NVDIMM vs SSD

NVDIMM integrates DRAM with non-volatile memory, typically NAND flash, on a single module.

NVDIMMs:

  • high performance and data persistency
  • Ideal for applications requiring fast data access and reliability

SSDs:

  • High storage capacity
  • Suitable for general data storage and applications with less stringent performance requirements
