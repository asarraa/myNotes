HDDs

Disks can be seen by an OS as a collection of data blocks that can be read or written
independently, each block is characterized by a unique numerical address called LBA (Logical Block Address). Typically, the OS groups blocks into clusters to simplify the access to the disk. Clusters are the minimal unit that an OS can read from or write to a disk. Typical cluster sizes range from 1 disk sector (512 B) to 128 sectors (64 KB).

File and clusters

Clusters contains:

  • File data: the actual content of the files
  • Meta data: the information required to support the file system.
    • File names
    • Directory structures and symbolic links
    • File size and file type
    • Creation, modification, last access dates
    • Security information (owners, access list, encryption)
    • Links to the LBA where the file content can be located on the disk

Deleting files

Deleting a file requires only to update the meta-data to say that the blocks where the file was stored are no longer in use by the OS.

Deleting a file never actually deletes the data on the disk: when a new file will be written on the same clusters, the old data will be replaced by the new one.

External fragmentation

As the life of the disk evolves, there might not be enough space to store a file contiguously. In this case, the file is split into smaller chunks that are inserted into the
free clusters spread over the disk.
The effect of splitting a file into non-contiguous clusters is called external
fragmentation. As we will see, this can reduce a lot the performance of an HDD.

The defragmention process tries to reorganize files in the storage (try to stare file bytes contiguously) in order to have better performances.

Hard disk drives anatomy

Hard disk drive (HDD) is a data storage using rotating disks (platters) coated with
magnetic material. Data is read in a random-access manner, meaning individual blocks of data can be stored or retrieved in any order rather than sequentially.
An HDD consists of one or more rigid ("hard") rotating disks (platters) with magnetic heads arranged on a moving actuator arm to read and write data to the surfaces.

Addressing and geometry

From the outside, a hard drive looks like a big block of data, broken up into small pieces called sectors. Each sector usually holds 512 or 4096 bytes of data. When you write to a sector, it happens atomically. However, if you're writing across multiple sectors and something interrupts the process (like a sudden power loss), only part of the data might make it to the disk. This is called a torn write.

Each sector also includes a bit of extra information â a header to identify it, and an error correction code to help detect and fix mistakes in the data.

Inside the drive, the structure is more physical. Data is written in circular paths called tracks, which are laid out like rings on each spinning disk, or platter. If a drive has multiple platters (which it often does), the tracks that line up vertically across them are called a cylinder. This helps organise and access data more efficiently.

The platters themselves are double-sided and spin constantly thanks to a motor, usually at speeds like 7200 or 15000 revolutions per minute (RPM). The spinning allows the read/write heads to quickly jump to the right location and access the data you need.

Caching

Modern hard drives include a small amount of built-in memory â usually just a few megabytes â known as the disk buffer or disk cache. This is a type of RAM inside the disk itself, and it's used to temporarily store data during read and write operations to boost performance.

Sometimes this memory is used in a very specific way, like a track buffer. For example, when the drive head reads from a track, it might grab the entire track and hold it in the buffer, even if only part of it was requested. This makes follow-up reads from nearby data much faster, because the disk already has that data in memory.

Read Caching

In read caching, the disk uses its buffer to store recently or frequently accessed data, so if itâs needed again soon, it can be served quickly â without waiting for the disk to spin or the head to seek. This helps reduce the delays caused by the mechanical parts of the drive.

Write Caching

For writes, disks can use the buffer to temporarily hold data before itâs actually written to the spinning platters. There are two main strategies:

  • Write-back caching: The disk tells the system that the data has been written as soon as it's stored in the cache â not when itâs actually saved to disk. This makes the system faster, but itâs risky: if power is lost before the data reaches the disk, that data is gone. Some drives reduce this risk by using disks with include flash memory for persistent caching (hybrid solutions like NVIDMM).
  • Write-through caching: In this safer approach, the disk only reports the write as complete after the data has actually been written to disk. Itâs slower, but more reliable â especially in case of crashes or power failure.

Common Disk Interfaces

Over time, disk interfaces have evolved significantly, improving performance, compatibility, and ease of use.

  • ST-506 (Shugart Technology 506):
    This was one of the earliest hard disk interfaces, introduced in the early 1980s. It required the operating system to address data using a CHS (Cylinder/Head/Sector) scheme.
    • This meant that to access any data, the system had to know the exact physical layout of the disk: which cylinder (a vertical group of tracks), which head (which platter surface), and which sector (a subdivision of a track).
    • Limitation: This made disk management complicated and inflexible, as any change in physical geometry (like using a bigger disk) would break compatibility unless manually reconfigured.
  • ATA / IDE (Advanced Technology Attachment / Integrated Drive Electronics):
    This standard revolutionized disk interfaces by integrating the disk controller onto the drive itself, instead of requiring a separate controller card.
    • It also abstracted away much of the CHS complexity from the user and the OS.
    • It supported LBA (Logical Block Addressing) partially, allowing systems to refer to disk sectors linearly rather than through physical geometry.
    • What it solved: No more manual handling of disk geometry, simpler installation, and lower hardware cost.
    • Use case: This became the default standard for consumer PCs in the 1990s and early 2000s.
  • SATA (Serial ATA):
    SATA improved over IDE/ATA in several critical ways:
    • It replaced parallel cables with serial communication, which allowed higher data transfer speeds, better reliability, and simpler, thinner cables that improved airflow in computers.
    • It fully embraced Logical Block Addressing, making it much easier for software to interact with disks without worrying about physical layout.
    • It introduced hot-swapping, meaning drives could be added or removed without shutting the system down (if supported by the OS).
    • What it solved: Performance bottlenecks from parallel communication (IDE), and physical limitations like bulky cables and complex connectors.
    • Use case: Became the standard for modern HDDs and SATA SSDs in desktops and laptops.
  • SCSI (Small Computer System Interface):
    SCSI was a more enterprise-focused technology that supported:
    • A packet-based communication system, similar to how networks operate (like TCP/IP), allowing efficient multitasking.
    • A larger number of devices on the same bus compared to ATA/IDE.
    • Internally, SCSI devices used LBA and translated it to their native geometry, providing full abstraction.
    • What it solved: SCSI enabled better multitasking, parallel command execution, and support for more complex storage configurations than ATA.
    • Use case: Widely used in servers, workstations, and enterprise systems, especially before the rise of SATA and SAS.
  • SAS (Serial Attached SCSI):
    SAS is the modern successor to parallel SCSI.
    • It replaced parallel signaling with serial communication, similar to SATA, offering higher speeds and better scalability.
    • It is backward-compatible with SATA, meaning SATA drives can be used in SAS environments (but not the reverse).
    • SAS supports features like dual-porting for redundancy, advanced queuing, and greater reliability, which are important for data centers and enterprise environments.
    • What it solved: Improved on all the physical and performance limitations of traditional SCSI while keeping compatibility and adding modern features.
  • Modern Transport-Independent Storage (USB, FireWire, Optical Media):
    These interfaces focused more on convenience and portability than raw speed.
    • USB drives and external HDDs/SSDs made it easy to carry data between systems without internal installation.
    • FireWire (IEEE 1394) provided faster data transfer than USB 1.0/2.0 and was popular in multimedia workflows.
    • Optical media (CDs, DVDs, Blu-rays) became common for data distribution, backups, and media playback.
    • What they solved: Provided plug-and-play external access to storage for backup, media, or transport, without needing to open up a computer.
  • iSCSI (Internet SCSI):
    iSCSI takes the SCSI protocol and transmits it over TCP/IP networks, allowing storage devices to be located remotely but accessed as if they were local disks.
    • This means that storage in a different part of the buildingâor even across the globeâcan be mounted and used by a server or computer as if it were connected directly.
    • What it solved: Eliminated the need for physical proximity between storage and computing systems; enabled the rise of SANs (Storage Area Networks).
    • Use case: Common in enterprise, cloud, and virtualized environments where centralized storage is critical.

Types of Delay with disks

There are four types of delay:

  1. Rotational delay: Time to rotate the desired sector to the read head, thus related to RPM
  2. Seek delay: Time to move the read head to a different track
  3. Transfer time: Time to read or write bytes
  4. Controller Overhead: Overhead for the request management

Disk Scheduling

Caching improves disk performance but cannot compensate for poor random access times. To optimise disk usage, requests can be reordered to reduce seek times based on their position on disk.

Key disk scheduling algorithms:

  • FCFS (First Come, First Serve): Processes requests in the order they arrive.
  • SSTF (Shortest Seek Time First): Picks the request closest to the current head position.
  • SCAN (Elevator Algorithm): Sweeps the entire disk and while doing so serves the requests. That is, moves the head in one direction, servicing requests, until the disk is over, then reverses the direction and serves the remaining ones until again the disk is over.
  • C-SCAN: Ciruclar SCAN. It is like SCAN but only services requests in one direction before returning to the start. That is, moves the head in one direction, servicing requests, until the disk is over. Than the head moves to the opposite end of the disk and doesn't serve request while doing so. At this point moves back to the starting point and serves requests while doing so.
  • C-LOOK: A variant of C-SCAN that only goes as far as the last request before resetting. That is, the same as C-SCAN but doesn't reach the end of the disk but only the last request.

Where scheduling is implemented:

  • OS-level: Reorders requests using logical block addresses (LBA), but canât optimize for rotational latency.
  • On-disk: Has full knowledge of head and platter positions, allowing better optimization, but depends on hardware and drivers.
  • Disk Command Queue (NCQ): Modern disks include a queue for pending I/O; the disk can reorder requests internally.
  • Joint scheduling (OS + disk): Can lead to conflicts or inefficiencies, as shown in studies like "NCQ vs. I/O Scheduler."

Q: [2.3] What is the role of HDDs in a computing node's storage infrastructure?
A: HDDs provide persistent storage by organizing data into addressable blocks and clusters, allowing efficient file access, storage, and deletion. They are a key component of node-level storage in server infrastructures.

Q: [2.3] What are logical blocks and clusters in HDDs?
A: Logical blocks (LBA) are the smallest addressable units on a disk, while clusters are groups of blocks used by the OS as the minimal units of I/O. Clusters can contain both file data and metadata.

Q: [2.3] What metadata does a file system store in clusters?
A: File metadata includes file names, directory structures, symbolic links, sizes, types, timestamps, security info, and links to the LBA for file data location.

Q: [2.3] What happens when a file is read from or written to disk?
A: Reading or writing occurs at the cluster level. The actual disk space used is rounded up to the nearest cluster size, which can lead to internal fragmentation.

Q: [2.3] What is internal fragmentation in HDDs?
A: Internal fragmentation is the wasted space resulting from a file not fully utilizing its allocated cluster space.

Q: [2.3] What does deleting a file on an HDD actually do?
A: Deleting a file updates metadata to mark its space as free, but the data remains until overwritten by new files.

Q: [2.3] What is external fragmentation on disks?
A: External fragmentation occurs when files are split into non-contiguous clusters, reducing read/write performance. Defragmentation reorganizes files to fix this.

Q: [2.3] What is the physical structure of an HDD?
A: HDDs consist of spinning platters with magnetic surfaces and read/write heads on actuator arms, allowing random-access data storage.

Q: [2.3] How is data organized inside an HDD?
A: Data is stored in sectors on concentric tracks. Matching tracks across platters form cylinders. Platters spin at constant RPMs, enabling access via moving heads.

Q: [2.3] What is a torn write in HDDs?
A: A torn write happens when a multi-sector write is interrupted (e.g., due to power loss), causing only part of the data to be written.

Q: [2.3] What is the purpose of the disk buffer or cache in HDDs?
A: Disk buffers temporarily store data during reads or writes to reduce delays from mechanical operations and improve performance.

Q: [2.3] How does read caching work in HDDs?
A: The disk cache stores recently or frequently accessed data so it can be retrieved quickly without needing to re-read from disk.

Q: [2.3] What are the types of write caching in HDDs?
A: Write-back caching acknowledges writes after buffering (faster but risky), while write-through caching waits for the data to be written to disk (safer but slower).

Q: [2.3] What are common disk interfaces and how have they evolved?
A: Disk interfaces evolved from ST-506 and ATA to SATA (faster, LBA-based) and SCSI/SAS (packet-based, multitasking). Modern transport includes USB, optical media, and networked storage like iSCSI.

Q: [2.3] What are the four types of delay in HDDs?
A: The delays are rotational (waiting for the sector), seek (moving the head), transfer (reading/writing data), and controller overhead (processing the request).

Q: [2.3] How is average rotational and seek delay calculated in HDDs?
A: Average rotational delay is half a rotation. Average seek delay is modeled as one-third of the maximum seek time due to variable distances and movement phases.

Q: [2.3] What is the full I/O service time in HDDs?
A: Service time includes seek time, rotational delay, transfer time, and controller overhead. Response time also includes queuing delay based on workload and disk performance.

Q: [2.3] What are common disk scheduling algorithms?
A: FCFS (first in, first out), SSTF (closest request first), SCAN/C-SCAN (elevator-style), and C-LOOK (optimized C-SCAN) reduce seek times by reordering requests.

Q: [2.3] Where can disk scheduling be implemented?
A: Scheduling can occur in the OS, within the disk itself, or jointly. OS-level uses LBA; disks use physical position. NCQ enables disks to reorder I/O internally.
