Access control

source

Reference Monitor: is the component which enforces access control policies ("who does what on which resource"). Present in all modern kernels.
Given a request, it has to find and evaluate the relevant security policy and verify the identity of the one who sent the request, in order to decide whether the access is granted or denied.

Access control models

Access control models define how actions are (un)assigned to subjects and objects
Can be divided in:

  • Discretionary Access Control (DAC)
  • Mandatory Access Control (MAC)
  • Role-Based Access Control (RBAC)

Discretionary Access Control (DAC)

The owner of a resource can discretionarily decide its access privileges.
All OSs implement DAC

DAC in UNIX

Subjects: users, groups
Objects: files (actually everything since in UNIX everything - also devices- is a file)
Actions: read, write, execute

In bash, the command ls -l lists files and directories in the current directory in long format, showing detailed information such as:

  • File type and permissions
  • Number of hard links
  • Owner
  • Group
  • File size
  • Modification date/time
  • File name

On permission triads

Permission triads are the way permission are represented. The three components refer to:

  • Permission of user
  • Permission of group
  • Permission of others

Permissions are represented with three letters:

  • r: read
  • w: write
  • x: execute

Each letter represent a flag: if there is the corresponding letter, it means the flag is true, if there is a dash the flag is false.

Now, a permission triad is formed by 3 groups of 3 letters. We can consider each flag as a binary digit (1 or 0), and consider each group as a binary number, notice the binary number can go from 0 to : it can be represented as a single base eight digit.

Example

Given the following permission triad represented in base eight: 755
rwx r-x r-x

General Model of DAC systems

We need to model the following entities:

  • Subjects: who can exercise privileges
  • Objects: entity on which privileges are exercised
  • Actions: which can be exercised

Protection state (triple S,O,A) for the entities in a system can be modeled easily with an access matrix with Subjects in the rows and Objects in the columns. represents privileges of subject on object .

Notice that this matrix is usually sparse, in fact usually just few users have privileges each object.

This kind of representation lead us to the HRU model. Here we have:

  • Basic operations:
    • CREATE (or DESTROY) subject
    • CREATE (or DESTROY) object
    • ADD (or REMOVE) into
  • Transitions: is a sequence of basic operations
    A possible example of transition is the one that allows a certain subject to create a file.
    It can be naively thought as:
    1. Create object
    2. Add "own" into
    3. Add "read" into
    However, in case the object already existed, this would cause the user to get privileges on it without the consent of the owner of . This tells us that this transition requires at least an if construct.
    Hence: transition can generate safety problems if not correctly designed.
Safety problems

Formally:
Given an initial protection state and set of transitions, is there any sequence of transitions that leaks a certain right (for which the owner is removed) into the access matrix?

If not, then the system is safe with respect to right

In a generic HRU model with infinite resources answering to that question is an undecidable problem.

Common DAC implementations

We said that the access matrix is usually sparse. Thus alternative implementations for HRU model are used:

  • Authorization table: records non-null (S,O,A) triples -> typically used in DBMSs

  • Access Control Lists (ACL): records by column (for each object, the list of subjects and authorizations)

  • Capability List: records by row (for each subject, the list of objects and authorizations)

A brief comparison between lists implementations:
Of course, ACLs are efficient for per object operations, while Capabilities are efficient for per subject ones.
Since the most common case is that object change and subject stay, ACLs are the most used ones, however in POSIX systems, the Capabilities are optionally available.
One disadvantage of ACLs is that they do not allow for multiple users (however this is partially achievable via groups).

DAC shortcomings

  • DAC cannot prove safety

  • DAC allow for access control on objects but not on the data inside objects (granularity). This means that if a user is granted read access to a file, they inherently gain access to the entire content of that file, regardless of whether all of it is relevant or appropriate for their role. this means:

    • Susceptible to malicious user problem: since users are allowed to make their own decisions about whom to grant access to their resources, they may inadvertently or deliberately allow unauthorized access
    • Susceptible to trojan horse problem: DAC permits any process executed by a user to inherit the access privileges of that user. This means that if a user runs a program containing hidden malicious code (a Trojan Horse), that program can act with the full authority of the user.
      The issue is illustrated by a classic example: suppose a trusted user (e.g., "Brown") runs a shared word processing application. An untrusted user (e.g., "Black") has embedded a Trojan Horse into that application. Once executed, the malicious component of the application silently reads sensitive information from a file to which Brown has legitimate access, such as an employee record. It then copies that data into another file that Black can access.

  • DAC suffers problems of scalability and management. In fact, each user-owner can potentially compromise security of the system with their own decisions.

Mandatory Access Control (MAC)

Idea: do not let owners assign privileges.
Privileges are set by a security administrator, he defines:

  • A classifications of subjects, called clearance
  • A classification of objects, called sensitivity

The classification is composed of:

  • a strictly ordered set of secrecy levels.
    Ex: Top secret > Secret > For Official Use Only > Unclassified
  • a set of labels, which are orthogonal to the classification:
    Ex: Policy, Energy, Finance, ATOMAL, NOFORN (no foregin parties)

Notice that the pairs {Secrecy Level, Label} form a lattice.

Bell-LaPadula Model (BLP)

This model defines two MAC rules:

  1. No read up (simple security property): A subject at a given security level is not permitted to read data at a higher security level. This ensures that users cannot access information that is more sensitive than their clearance level.
  2. No write down (star property): A subject at a given security level is not permitted to write to data at a lower security level. This rule prevents sensitive information from leaking to less secure areas, either accidentally or maliciously.

In addition to these MAC rules, the model includes a Discretionary Access Control (DAC) component:

  1. Discretionary Security Property: states the use of an access matrix to specify the discretionary access control. That is, it provides an additional layer of control by specifying what kind of access (e.g., read, write) is actually permitted. Notice that the access matrix is used only after MAC checks have passed. If MAC denies the access (e.g., you try to read a file at a higher level), the DAC rules are not even consulted.

The Bell-LaPadula model is based on the tranquillity property, that states that: secrecy levels of objects cannot change dynamically during system operation.
The consequence is there is a single flow of information toward higher secrecy levels: subjects may read from or write to data at their own level or higher (in the case of reading), but are restricted from leaking information to lower levels.

it does not prevent unauthorized or improper modification of information. For systems where integrity is critical, such as in commercial or financial environments, models like Biba are more appropriate, as they are explicitly designed to enforce integrity constraints.
