Web Application Security

The Evolution and Vulnerability of Web Applications

In recent years, we've witnessed a profound shift in how software is delivered. Traditional desktop applications are increasingly giving way to web applications—software that runs within a browser and communicates with remote servers. This shift affects everything from corporate intranets to consumer-facing services offered through Software-as-a-Service (SaaS) and cloud platforms.

This evolution makes sense from a business perspective: web applications are easier to update, simpler to maintain, and enable subscription models that provide steady revenue. However, this convenience comes with significant security challenges. Why? Because web applications are often publicly exposed on the internet and built atop HTTP—a protocol that was never designed with state management or robust authentication in mind. These critical security features had to be retroactively grafted onto HTTP, creating inherent vulnerabilities.

Understanding the Web Application Architecture

To grasp the security challenges, let's first understand how web applications typically work. Most follow a three-tier architecture:

First Tier: HTTP Client (Browser) - This is the interface through which users interact with the application. The browser renders HTML, executes JavaScript, and manages user interactions.

Second Tier: HTTP Server and Application Server - The HTTP server handles web requests and communicates with the application server, which contains the "business logic"—the rules and workflows that make the application function.

Third Tier: Database Server - This stores persistent data that the application needs to function, from user credentials to application content.

When you visit a web page and click "login," your browser sends a request like /login.php?user=foo to the server. The server processes this request, checks credentials against the database, and responds accordingly.

The Fundamental Disconnect: Developer vs. Attacker Mindset

Here's where the trouble begins: web developers naturally see the client as a cooperative part of the application. They envision users clicking buttons, filling forms, and interacting with the interface as designed. This creates a dangerous blind spot.

Attackers don't play by these rules. They understand that HTTP is simply text-based communication between computers. Using tools like cURL, they can craft custom HTTP requests that bypass the interface entirely:

$ curl -v http://httpbin.org/get
> GET /get HTTP/1.1
> Host: httpbin.org
> User-Agent: curl/7.43.0
> Accept: */*

They can easily modify headers:

$ curl -v -H 'User-Agent: foobar-agent/7.43.1' http://httpbin.org/get
> GET /get HTTP/1.1
> Host: httpbin.org
> Accept: */*
> User-Agent: foobar-agent/7.43.1

Or send custom data:

$ curl --trace-ascii /dev/stdout --data user=foobar http://httpbin.org/post
0000: POST /post HTTP/1.1
...
0000: user=foobar

This leads us to the golden rule of web application security: The client is never trustworthy. Any security measure implemented on the client side (like JavaScript validation of form inputs) can be completely bypassed by an attacker who crafts HTTP requests directly.

The Science and Art of Input Filtering

Since we can't trust client-side data, we must rigorously filter all inputs on the server side. But this is surprisingly difficult to get right. Effective filtering involves three progressive layers of protection:

  1. Allowlisting (formerly "whitelisting"): This means only permitting input that matches expected patterns. For example, if we expect a phone number, we might only allow digits and certain formatting characters. This is the strongest approach because it rejects anything not explicitly permitted.
  2. Blocklisting (formerly "blacklisting"): This involves rejecting known-dangerous patterns. While generally less secure than allowlisting, it can provide an additional security layer.
  3. Escaping: This transforms potentially dangerous characters into safer equivalents, making them display correctly without executing as code.

To understand why this is challenging, let's examine two concrete examples:

Filtering a Phone Number Input

For a phone number field, we might start with an allowlist accepting only digits. However, this approach would reject valid international formats like "+39-555-1234." A better allowlist would permit numbers, plus signs, hyphens, and spaces.

In this case, we probably don't need additional blocklisting or escaping, as phone numbers typically don't contain characters that can trigger code execution when properly processed by the application backend.

Filtering Blog Comments

Now consider a more complex scenario: a blog where users can post comments. If we strictly limit input to alphanumeric characters and basic punctuation, we'd create a secure system—but at what cost? Users couldn't share mathematical formulas (which need symbols like < and >), code snippets, or even basic formatting.

The intuitive solution might seem to be blocking dangerous elements like <SCRIPT> tags. But this opens a Pandora's box of security problems, as we'll soon discover.

Cross-Site Scripting (XSS): The Shape-Shifting Threat

Cross-Site Scripting (XSS) is one of the most persistent and dangerous vulnerabilities in web applications. It occurs when an application allows client-side code (typically JavaScript) to be injected into a page, which then executes in visitors' browsers.

To understand why this is dangerous, imagine a simple blog application that allows comments. Without proper filtering, an attacker could post a comment containing:

<SCRIPT>
alert('JavaScript Executed');
</SCRIPT>

When other users view the page containing this comment, the script executes in their browsers, displaying a popup. While a popup seems harmless, the same technique could be used to steal cookies, hijack sessions, or perform actions on behalf of the victim.

XSS attacks come in three distinct varieties, each operating through different mechanisms and requiring different defensive strategies. Understanding these differences is crucial for effective protection:

1. Stored (Persistent) XSS: The Silent Threat

Stored XSS represents the most dangerous form of cross-site scripting because the malicious code becomes a permanent part of the vulnerable website. Here's how it typically unfolds:

  1. Injection Phase: An attacker identifies a website feature that stores user input in a database—such as comment sections, user profiles, forum posts, or product reviews.
  2. Storage Phase: The attacker submits content containing malicious JavaScript, which passes through inadequate filters and gets stored in the application's database.
  3. Dormant Phase: The malicious code sits quietly in the database, waiting for victims. Unlike other attacks that require active distribution, stored XSS patiently waits for normal user traffic.
  4. Execution Phase: When any user (including administrators with elevated privileges) views the affected page, their browser automatically loads and executes the malicious JavaScript.

Stored XSS is particularly devastating because:

  • It affects all visitors to the compromised page, potentially reaching thousands of users
  • It persists until specifically removed from the database, potentially lasting years
  • It executes within the security context of the legitimate site, bypassing many protections
  • It affects users during normal browsing, with no suspicious actions required on their part
  • It often executes with the user's authentication level, potentially compromising admin accounts

Real-World Example: In 2005, the social networking site MySpace suffered a devastating stored XSS attack known as the "Samy Worm." A user named Samy Kamkar created a profile containing malicious JavaScript that would add him as a friend, copy itself to the victim's profile, and display "but most of all, Samy is my hero." Within 20 hours, over 1 million users had unknowingly been infected, making Samy a "friend" to almost everyone on MySpace and ultimately forcing the site to temporarily shut down.

2. Reflected (Non-Persistent) XSS: The Deceptive Link

Reflected XSS differs from stored XSS in that the malicious code isn't permanently stored on the server but is instead "reflected" back to the user in the immediate response to a request. Here's the attack process:

  1. Crafting Phase: The attacker identifies a vulnerable page where user input from the request (such as query parameters, form fields, or HTTP headers) is reflected in the response without proper sanitization.
  2. Distribution Phase: Unlike stored XSS, the attacker must actively distribute the attack vector—typically through misleading links sent via email, social media, or other websites.
  3. Deception Phase: The malicious link often appears legitimate or enticing, such as https://trusted-site.com/search?q=<script>evil_code()</script> or disguised with URL shorteners.
  4. Execution Phase: When a victim clicks the link, their browser sends the request to the vulnerable website, which then reflects the malicious code back in its response, executing in the victim's browser.

The technical mechanism often involves server-side code that directly outputs request parameters:

<?php
// Vulnerable code that reflects 'query' parameter directly in the response
$searchTerm = $_GET['query']; 
echo "Search results for: " . $searchTerm;  // Dangerous! No sanitization!
?>

When a user visits a URL like https://vulnerable-site.com/search.php?query=<script>stealCookies()</script>, the server responds with:

Search results for: <script>stealCookies()</script>

The browser then executes the script, not realizing it's malicious.

Reflected XSS is particularly insidious because:

  • It can bypass security measures that focus only on stored content
  • It leaves minimal evidence on server logs, making it harder to detect
  • It often abuses user trust in legitimate websites
  • It can be widely distributed through social engineering campaigns

Real-World Example: In 2018, researchers discovered a reflected XSS vulnerability in PayPal's checkout page. The vulnerability allowed attackers to create legitimate-looking PayPal checkout links that, when clicked, would execute arbitrary JavaScript in the victim's browser while on the authentic PayPal domain. This could have been used to steal payment details or hijack accounts.

3. DOM-Based XSS: The Client-Side Vulnerability

DOM-based XSS represents the most subtle and technically complex form of cross-site scripting. Unlike the other types, it occurs entirely within the client's browser and never involves sending the malicious payload to the server. Here's how it works:

  1. Identification Phase: The attacker finds a web application that uses client-side JavaScript to manipulate the Document Object Model (DOM) based on user-controllable data, such as URL fragments (content after the # symbol), which browsers don't send to servers.
  2. Exploitation Phase: The attacker crafts a URL with malicious content in a location that will be processed by the client-side script but never sent to the server.
  3. Execution Phase: When the victim loads the page, the legitimate JavaScript on the page unwittingly takes the attacker's input and executes it or inserts it into the DOM in an unsafe way.

The vulnerability typically exists in code that uses unsafe DOM manipulation methods with user-controllable input:

// Vulnerable code that unsafely uses the URL fragment
var pos = document.URL.indexOf("fragment=") + 9;
var fragment = document.URL.substring(pos);
document.write("<div>You selected: " + fragment + "</div>");

When a user visits https://vulnerable-site.com/page.html#fragment=<img src=x onerror=alert(1)>, the JavaScript grabs the fragment and inserts it directly into the DOM, executing the malicious code.

What makes DOM-based XSS particularly challenging:

  • It completely bypasses server-side security controls since the payload never reaches the server
  • It can't be detected by server-side logging or monitoring
  • It often exploits modern JavaScript frameworks and libraries that manipulate the DOM extensively
  • It can affect otherwise secure applications that have robust server-side protections
  • It frequently involves multiple steps of processing and encoding/decoding, making it harder to track

Real-World Example: In 2019, a security researcher discovered a DOM-based XSS vulnerability in Gmail's AMP4Email feature. The vulnerability existed in the client-side JavaScript that rendered dynamic email content. By crafting a special email with malicious AMP content, an attacker could potentially execute JavaScript in the context of Gmail when the victim opened the email, potentially accessing their emails or sending messages on their behalf.

Comparison of XSS Types

Understanding these distinctions helps developers implement appropriate defenses at each layer of the application stack and recognize the specific vulnerabilities their code might contain.

Why XSS Matters: Beyond Pop-up Alerts

Developers sometimes dismiss XSS vulnerabilities as minor annoyances that merely result in pop-up alerts. This dangerous misconception ignores the true power of XSS attacks, which can:

  • Steal cookies and hijack sessions: An attacker can use XSS to access a victim's cookies, potentially including session identifiers that grant access to their account.
  • Manipulate active sessions: Once inside a user's browser context, attackers can perform actions on the user's behalf—transferring funds, changing settings, or accessing sensitive information.
  • Access private information: Any data visible to the legitimate user in their browser becomes accessible to the attacker's script.
  • Facilitate drive-by downloads: XSS can be used to trigger downloads of malware without user awareness.
  • Bypass the Same-Origin Policy: Perhaps most concerning, XSS effectively circumvents one of the browser's core security mechanisms.

The Same-Origin Policy: A Critical Security Boundary

The Same-Origin Policy (SOP) is a fundamental security mechanism implemented by all web browsers. It dictates that client-side code (like JavaScript) loaded from origin A should only be able to access data from origin A—where an "origin" is defined by the combination of protocol, host, and port.

For example, JavaScript running on https://bank.com shouldn't be able to access data from https://mail.com or even http://bank.com (different protocol).

This policy creates crucial security boundaries between websites, preventing malicious sites from accessing your data on legitimate sites. However, the modern web has increasingly blurred these boundaries through:

  • Cross-Origin Resource Sharing (CORS): A mechanism that allows controlled access across origins
  • Client-side extensions: Browser extensions that may have elevated privileges
  • Embedded frames and third-party content: Which introduce complex security considerations

Most importantly, XSS attacks can bypass SOP entirely because the malicious code executes within the origin of the vulnerable site, inheriting all its privileges.

The Futility of Blocklisting: A Security Arms Race

Now, let's return to our blog comment filtering problem. Why not simply block known dangerous elements like <SCRIPT> tags?

This approach quickly devolves into an unwinnable arms race against increasingly creative bypass techniques. Consider these evasion methods:

1. Alternative Execution Vectors

Blocking <SCRIPT> isn't enough because numerous HTML elements can execute JavaScript:

<img src="/doesnotexist" onerror="alert('JS!');">
<svg onload=alert('JS Executed');">

HTML has dozens of event handlers like onerror and onload that can execute scripts.

2. The JavaScript URI Scheme

JavaScript can be executed through the javascript: URI scheme:

<iframe src="javascript:alert('JavaScript Executed');">
<a href="javascript:alert('JS Executed');">click here!</a>

3. Obfuscation Through Whitespace

Browsers are remarkably forgiving with HTML parsing, often ignoring whitespace that would invalidate filters:

<iframe src="javas
cript:alert('JavaScript Executed');">

4. HTML Entity Encoding

Attackers can encode characters using HTML entities to bypass filters:

<iframe src="javasc&#09;ript:alert('JavaScript Executed');">
<iframe src="javasc&#X0A;ript:alert('JavaScript Executed');">
<iframe src=javasc&#000010;ript:alert('JavaScript Executed');>

The browser decodes these entities (&#09; is a tab, &#X0A; and &#000010; are line feeds) before processing the HTML.

5. Browser-Specific Quirks

Some attacks work only in specific browsers due to implementation quirks:

<img src="&{alert('JavaScript Executed')};">

6. Filter Chain Exploitation

Particularly sophisticated attackers might combine techniques to exploit how filters interact:

<iframe src="java&{script{alert('JavaScript Executed')};">

If a filter removes &{ characters but doesn't recheck the result, the remaining text becomes the executable javascript:alert(...).

7. Style-Based Attacks

Even CSS can be weaponized in older browsers:

<style type="text/javascript"> ... </style>

<p style="left:expression(eval('alert(\'JavaScriptExecuted\');window.close()'))">

<style type=text/css>
@import url(javascript:alert('JavaScript Executed'));
</style>

This endless parade of bypass techniques demonstrates why blocklisting is fundamentally flawed: you can never anticipate every possible attack vector.

The Right Approach: Context-Aware Escaping

Instead of blocking specific patterns, the correct approach is to transform potentially dangerous characters into their HTML-safe equivalents:

  • < becomes &lt;
  • > becomes &gt;
  • & becomes &amp;

This escaping ensures that special characters display correctly without being interpreted as HTML or JavaScript. Modern web frameworks often provide context-aware escaping functions that automatically apply the appropriate transformations based on where the data is being used (HTML context, JavaScript context, CSS context, etc.).

Building a Security Mindset for Web Development

To develop truly secure web applications, developers must embrace these principles:

1. Never trust client-side data - All data from the client, including not just form inputs but also cookies, headers, and URL parameters, should be considered potentially malicious.

2. Apply defense in depth - Use multiple layers of protection, including allowlisting, appropriate escaping, and server-side validation.

3. Use context-appropriate security measures - Different types of data and different parts of your application require different security approaches.

4. Keep security knowledge current - Web security is an evolving field, and new vulnerabilities emerge regularly.

5. Implement security at the architecture level - Security should be a fundamental consideration in application design, not an afterthought.

By approaching web application security with appropriate skepticism toward client-provided data and a deep understanding of attack vectors, developers can build resilient systems that protect both the application and its users from evolving threats. Rather than engaging in an endless battle against specific attack techniques, focus on fundamental security principles that address the root vulnerabilities in web architecture.

SQL Injection: The Database Vulnerability

SQL injection remains one of the most prevalent and dangerous attacks against web applications. At its core, this vulnerability occurs when user input is improperly handled before being incorporated into SQL queries, allowing attackers to manipulate the structure of those queries.

Understanding the Vulnerability

Consider a typical login page where users enter their username and password. Behind the scenes, the application might use code like this:

public void onLogon(Field txtUser, Field txtPassword) {
    SqlCommand cmd = new SqlCommand(String.Format(
        "SELECT * FROM Users WHERE username='{0}' AND password='{1}';",
        txtUser.Text, txtPassword.Text));
    SqlDataReader reader = cmd.ExecuteReader();
    if(reader.HasRows())
        IssueAuthenticationTicket();
    else
        RedirectToErrorPage();
}

When a legitimate user enters their credentials (for example, username "s.zanero" and password "s3cr3t!"), the application generates a SQL query:

SELECT * FROM Users WHERE username='s.zanero' AND password='s3cr3t!';

If the database contains a matching record, the user is authenticated. However, a malicious user can exploit this implementation by manipulating the input fields.

Authentication Bypass

An attacker might enter s.zanero';-- as the username. The resulting query becomes:

SELECT * FROM Users WHERE username='s.zanero';--' AND password='';

In SQL, the double dash (--) indicates a comment, causing the database to ignore everything that follows. This transforms the query to simply check if the username exists, completely bypassing the password verification.

Even more concerning, if an attacker doesn't know a valid username, they could use ' OR '1'='1';-- as input:

SELECT * FROM Users WHERE username='' OR '1'='1';--' AND password='';

Since '1'='1' is always true, this query will return all user records, granting the attacker access regardless of credentials.

Data Extraction

SQL injection isn't limited to authentication bypass. Attackers can also use these techniques to extract sensitive data. For example, with a query that displays user information:

SELECT name, phone, address FROM Users WHERE Id='userinput';

An attacker could inject:

' OR '1'='1';--

This would display all records in the Users table.

More sophisticated attacks can extract data from different tables using UNION statements:

' UNION ALL SELECT name, creditCardNumber, CCV2 from CreditCardTable;--

This technique works when the number and data types of columns match between the tables being combined.

Insertion Attacks

SQL injection vulnerabilities also exist in INSERT operations. For example, in an application that records exam results:

INSERT INTO results VALUES (NULL, 'userinput', '18')

An attacker could manipulate this to:

INSERT INTO results VALUES (NULL, 's.zanero', '30L') --', '18')

This allows unauthorized grade modification. Even more concerning, an attacker could insert multiple records:

INSERT INTO results VALUES (NULL, 's.zanero', '30L'), (NULL, 'f.maggi', '30L') --', '18')

Or extract sensitive information from other tables:

INSERT INTO results VALUES (NULL, 's.zanero', (SELECT password from USERS where user='admin')) --', '18')

Blind SQL Injection

Some applications don't display the results of SQL queries directly, making traditional SQL injection more difficult. In these cases, attackers might use "blind" SQL injection techniques, where they infer data based on the application's behavior rather than direct output.

Preventing SQL Injection

Several approaches can mitigate SQL injection risks:

  1. Input Validation and Filtering: Implement strict validation on all user inputs. For usernames, a whitelist approach might permit only alphanumeric characters and specific symbols like periods. For passwords, careful consideration is needed to balance security with usability.
  2. Prepared Statements (Parameterized Queries): Instead of building SQL strings through concatenation, use prepared statements with parameter placeholders:
$stmt = $db->prepare("SELECT * FROM users WHERE username = ? AND password = ?");
$stmt->execute(array($username, $psw));

This approach ensures that user input is treated as data, not executable code.

  1. Table and Field Naming Conventions: Avoid using table names that might reveal sensitive information about your database structure.
  2. Principle of Least Privilege: Limit database permissions according to each application's needs, reducing the potential impact of successful injections.

Code Injection Problems

The fundamental challenge in web security is the conflict between two requirements:

  1. The functional requirement to mix code (such as HTML) with data (such as user comments)
  2. The security requirement to never mix code and data

Whenever a parsing routine (like a browser's JavaScript parser) interprets control sequences found in user data, security vulnerabilities can emerge.

Information Leakage

Seemingly harmless information can often provide attackers with valuable insights into your system's structure or operation.

Error Messages

Detailed error messages can improve user experience but might reveal sensitive information. For example, distinguishing between "user not found" and "password incorrect" in login failures might seem helpful but actually assists attackers in enumerating valid accounts.

Debug Information

Debug traces accidentally left active in production environments can expose critical information such as:

  • Server and application versions
  • Database names, structures, and credentials
  • File system paths

User-supplied data included in error messages can also create reflected cross-site scripting (XSS) vulnerabilities.

URL Parameter Tampering

Web applications often use URL parameters to control their behavior. When these parameters are inadequately validated, attackers can manipulate them to access unauthorized resources or execute unintended operations.

Directory/Path Traversal

Path traversal attacks exploit insufficient validation of file paths. By using sequences like ../../../etc/passwd (possibly URL-encoded), attackers might access files outside the intended directory structure.

Password Security

Proper password handling remains crucial for web application security:

  1. Never store passwords in plain text. Use salted hashing to protect against both direct disclosure and rainbow table attacks.

  2. Design password reset mechanisms carefully:

    • Sending reset links to registered email addresses is generally acceptable
    • Sending temporary passwords via email or relying solely on security questions is problematic

  3. Implement bruteforcing protection, but consider implications carefully:

    • Simply locking accounts after N failed attempts can enable denial-of-service attacks
    • IP-based blocking is complicated by NATs, proxies, and shared IP addresses
    • Making account enumeration difficult helps prevent targeted attacks

Session Management

HTTP is inherently stateless, requiring mechanisms like cookies to maintain user state across requests.

Session Creation and Authentication

A typical session establishment flow involves:

  1. The user submits credentials to the web server
  2. The server validates these credentials, generates a unique session ID, and associates it with the user
  3. The server returns a cookie containing this session ID
  4. The browser stores this cookie and includes it with subsequent requests

Security Considerations for Sessions

Effective session management requires attention to several factors:

  1. Session ID Security: Session IDs must be unpredictable and resistant to brute-force attacks. They should have reasonable expiration periods set on the server side (not just in the cookie).
  2. Concurrency Handling: The application must correctly manage situations where multiple users interact with it simultaneously.
  3. Session Termination: Define clear policies for when and how sessions expire, and how to handle requests with expired session IDs.
  4. Session Storage: Consider performance and architectural implications of where session data is stored, particularly in distributed environments with multiple servers.

Cross-Site Request Forgery (CSRF)

CSRF attacks force authenticated users to execute unwanted actions without their knowledge. These attacks exploit the fact that browsers automatically include cookies with every request to a site where the user is authenticated.

CSRF Example

Consider a bank transfer form:

<form method="POST" action="/transfer.php">
  <h3>Transfer money</h3>
  Recipient: <input type="text" id="inpUser" name="to">
  Amount: <input type="number" id="inpAmount" name="amt">
  <button type="submit">Confirm</button>
</form>

An attacker could create a malicious page containing:

<form id="evil" style="display: none;" action="https://ironbank.7k/transfer.php" method="POST">
  <input type="hidden" value="50000" name="amt">
  <input type="hidden" value="S.Zanero" name="to">
  <input type="submit">
</form>
<script>document.evil.submit();</script>

If a user who is authenticated with the bank visits this malicious page, their browser will automatically submit the form with their authentication cookies, potentially transferring money without their consent.

CSRF Mitigation

  1. CSRF Tokens: Include unique, session-specific tokens in forms:
<form class="form-signin" method="post">
  <h3 class="form-signin-heading">Transfer money</h3>
  <input type="number" id="inputAmount" name="amt" class="form-control" placeholder="Amount" required>
  <input type="hidden" value="9GiKZU6HoR" name="csrf_token">
  <button class="btn btn-lg btn-primary btn-block" type="submit">Confirm</button>
</form>

The server validates that the token in the submitted form matches the one associated with the user's session.

  1. SameSite Cookies: Modern browsers support the SameSite cookie attribute:
    • SameSite=strict: Cookies are never sent for cross-site requests
    • SameSite=lax: Cookies are sent for cross-domain navigation but not for cross-site forms, images, frames, etc.

Session Hijacking

Since HTTP is stateless, session hijacking can occur through various means:

  • Stealing cookies via XSS attacks
  • Brute-forcing weak session IDs

Proper session management and security controls are essential to prevent these attacks.

Conclusion

Web application security requires a multi-layered approach addressing input validation, authentication, session management, and output encoding. By understanding common vulnerabilities and implementing appropriate safeguards, developers can significantly reduce the risk of successful attacks against their applications.

  • Google Gruyere (Web security)
  • Websec.fr (Web security)
  • pwnable.kr (Binary exploitation)
  • OverTheWire Bandit and Leviathan (Various security topics)
  • PicoCTF (Mixed security challenges)

With diligent attention to security principles throughout the development lifecycle, web applications can provide both rich functionality and robust protection against malicious attacks.
