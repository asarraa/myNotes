Summary criptography

Criptography (or criptology) is the discipline that studies the secure exchange of information in presence of attackers

General setting:

  • sender
  • receiver
  • encoder / decoder
  • plaintext
  • ciphertext

A cryptosystem is defined by a tuple (P, C, K, E, D):

  • plaintext space: string over an alphabet and lenght l
  • Chipertext space: string over an alphabet and lenght l'
  • key space: string over an alphabet and length lamda
  • Encryption function: PxK-->C
  • Decription function: CxK-->C

Cipher: is an implementation of the encryption and decryption function, so that D(E(p,k), k')=p.
If k=k' then we say the cipher is symmetric.

Security requirements

Kerckhoff's principles:

  1. ...
  2. a cipher must be secure even if the cryptosystem is known to the enemy.
  3. ...

In general, we have to assume the attacker has access to the cryptosystem, in particular it has access to:

  • P
  • C
  • K
  • Can call E and D

What do we want to protect?

  • Confidentiality: only the sender and the reciver can read to the plaintext
  • Integrity: no one can tamper the ciphertext
  • Authenticity: data and their origin are certified
  • Non-repudiation: the creator of a resource cannot negate its creation
  • Other features

Symmetric ciphers

Confidentiality

Perfect cipher definition: cipher perfect iff forall ptx and ctx P(ptx_sent=ptx)=P(ptx_sent = ptx | ctx_sent = ctx). That is, the fact that the attacker knows which sent has been sent (by eavedropping) it does not have additional information.

Pratical definition: a cipher with |P|=|C|=|K| is perfect iff

  1. each key has probability 1/|K| of being chosen, that is uniform distribution over the key space
  2. each key maps a single plaintext to one and only one ciphertext: given ptx and ctx, there exists only one key k so that E(ptx, k)=ctx

One cipher was developed: One-Time-Pad: ptx xor k = ctx.

  • Of course the key should be sampled with unifor random distribution.
  • Works as long k its not repeated , otherwise by performing operations (xor) between different ctx generated with the same k, it is possible to see some part of the ptx.
  • key long as plaintext

It is not vulnerable to brute-force attacks.

Problems:

  • k is too long and thus difficult to share to the receiver
  • difficult to generate random strings
  • too many keys needed, practically difficult to guarantee non repetition

Nash demostrated that a secure cipher is a cipher which is computationally hard to break. With hard we mean that the problem of decrypting a message of a secure cipher could be reduced to solving a NP problem, which under the assumption of P!=NP is impossible

Problem of the perfet cipher derive for the generation and distribution of a key long as the plaintext.
Idea: exchange a short key and use a deterministic function that taken this key of lenght lambda outputs a longer key of length lamda + n. This function goes under the name of Pseudorandom Number Generator (PRNG)

Since the key is shorter than the plaintext, the cipher is not perfect. However, if breaking the PRNG was hard, we can say the cipher is secure.
In particular, assuming the attacker can only do computations with polinomial complexity wrt lamda (length of the key) we can introduce:

CSPRNG: is a PRNG whose outputs cannot be distinguished from a random string over in

We build CSPRNG by using building blocks, called block ciphers.
Block cipher: is an encryption algorithm that takes as input a key k and a string ptx of a fixed length b and outputs a string of the same length as the input.
Given k, the encryption function E(k, ptx) corresponds to one of the possible permutations of the input.
This building block goes under the theoretical name of Pseudorandom permutation (PRP)

When is a block cipher said to be broken? If, without knowing the key, the plaintext can be derived from the ciphertext in less than operations.
That is, if the fastest way to do it is by bruteforcing the key (2 possible choices for every positions).

Famous block ciphers: DEA, DES --> deprecated, AES --> standard: block of 128 bits and key can be 128-192-256.

Now, we said that block ciphers operate on fixed block length. We could imagine to use directly block cipher to encrypt our message, in fact we see they can encrypt a plaintext that has length different from the one of the key. However, two cases:

  • plaintext < block size
  • plaintext > block size
    In the first case we can simply insert padding
    In the second case, we can think to split the plaintext and use several block ciphers. Problem: block ciphers are deterministic, so if there are repetition in the plaintext (as happens in pixels in pictures), the ciphertext could reveal patterns, so confidentiality would be compromised. This setting is called electronic codeblock (ECB)

Let's go back and use those block ciphers for the goal we intended:
by taking several block ciphers and input them with the same key and incremental counter as plaintext, they will all output a different ctx (this is called counter mode (CTR). If we put all those outputs together, we have a long string which can be seen as a stretched key. We have a PRNG and if the block cipher are not broken, we have a CSPRNG. Now we can use the stretched key to XOR the plaintext as we would have done in the perfect cipher. Actually we got several shorter strings of length a block, so we don't need to put them together, we can use them individually to XOR each of them with a substring of length b of the plaintext.

We said that for a block cipher to be considered secure, it should not breakable in less than . However, the proof of this would imply N=NP. So block ciphers are result of public contests.

Change of assumption: known-plaintext attacks.
If the block cipher is deterministic, the attacker which knows plaintext - ciphertext mapping, would be able to break the cipher (can infer which plaintext corresponds to which ciphertext).
To be able to resist to those attacks, non deterministic ciphers are needed. To do so, we can:

  1. Randomize the encryption
  2. Using nonce: for the CTR, pick a NONCE (number used once) as counter starting point. NONCE is public.
  3. Rekeying --> Symmetric ratcheting: mechanism to generate a different key for every message. Take a seed, put it into a PRNG, obtain a string which is two times the input, use the first half as a first key, put the second half as input to a second PRNG

Integrity

Setting: active attack --> the attacker can actively intercept and tamper with the sent data (es: bit-flipping attack).

Idea: attach to the message a tag that the attacker is not able to forge and can guarantee the message has not been tampered with. This tag is called MAC (message authentication code).

We need two functions:

  • compute_MAC(string, key) --> tag
  • verify_MAC(string, key) --> true/false

How to implement this? two approaches

  1. CBC-MAC: we use as tag an encrypted version of the message. We use the block ciphers to do it. We split the string in blocks and use them as input for each block cipher XORed with another string which is the output of the previous block cipher. The key is always the same. The final block's output will be the tag we will attach to the message.
    In practice: sender sends string + tag (computed as above), receiver will decrypt the message and compute the tag on it, veryfing it is the same as the one received. If it is, then the message has not been tempered with (attacker changes data --> MAC will not match, attacker changes data and MAC --> to do it consistently, he should know the key).
    Problem: inefficient since we need to read the file completely to compute a MAC.

HTTP servers use MAC in order to prevent users to modify cookies. Server compute MAC(cookie), then send both the cookie and the MAC tag to the browser. When the user contacts the server, it has to serve cookie and tag, the server computes MAC(cookie) again and check if it is the same as the tag sent by the user, in order to check no tampering with was done.

  1. Cryptographic Hash Functions. Is a function H: --> . That is takes a string of arbitrary length to a string of fixed length. To be a cryptographic hash function. the following problems must be computationally hard:
    • Invert the function -->ideally
    • Given a string s and it's hash H(s), find r!=s so that H(r) --> ideally
    • Collision: find r, s with r!=s such that H(s)=H(r) --> ideally (less because of birthday paradox)

Characteristics:
- Output is called digest.
- Since arbitrary input dimension and fixed output dimension, |Possible inputs| > |Possible outputs|, then the function is not injective (must be that multiple input are mapped to the same output)
- Minimum input dimension (to ensure) is called entropy.
- First and second problem are what differentiate an hash function from a cryptographic hash function.
- The second problem can be reduced to the first --> same complexity. However, as for the CTR block cipher, we cannot prove the ideal complexity are respected, since providing this kind of lower bound to algorithms would prove N=NP

Types of cryptographic hash functions:

  • SHA1 --> deprecated
  • SHA2 --> ok but sus (share some characteristic with broken functions)
  • SHA3 --> ok

How are cryptographic function used? In general, for fingerprints or to avoid storing password in clear.
In our case? We could think of just computing the MAC tag using hash functions, instaed of all the CBC-MAC stuff. In this case however if sender and users exchange msg and tag, since the tag is output of an hash function, which does not require a key, then everybody can tamper with the message, create a new tag accordingly.
Instaed, the new approach is: h(msg+key), that is, the tag is the digest of the msg concatenated to the key. So the receiver once decripted the message, can concatenate it with the key, apply h(msg+key) and check if the obtained tag is the same. More efficient than CBC-MAC since fixed length tag which don't require full scan of the msg.

Standardized way to use hash function to provide MAC: HMAC

Asymmetric ciphers

Asymmetric ciphers are ciphers where encryption and decryption keys are different.
They were invented because:

  • symmetric ciphers can't provide confidentiality
  • exchange the key its difficult --> recursive encryption problems
    Problem of asymm ciphers --> they are slower

How do they work? Each endpoint has a pair of key (kpri, kpub). The public one is generated from the private one through a function: kpub=f(kpri). Of course the cipher has some features:

  • Computationally hard to derive kpri from kpub (f not invertible)
  • public (private) key encryption --> private (public) key decription
  • If the message was encrypted with public key, computationally hard to decript it without the private key.

Again, we cannot prove lower bounds --> public contests

Confidentiality

  • B sends its public key to the A
  • A encrypts message with B' public key, B decrypts with its private key

For bi-directional communication, A should generate its key pair and send its public key to B.
Since asymmetric encryption is slow, it is usually used only for exchanging symmetric key --> key encapsulation

  • just one pair is enough, no bi-directional communication needed
  • don't need to consider known-palintext attacks for asymmetric ciphers (I think because we use same techniques because we use symmetric ciphers for actual communication)

Standard cryptosystem --> RSA

Integrity

We use MAC as in the symmetric case

Authenticity

We use signatures

  1. A sends msg in clear
  2. A sends E(msg, private key)
  3. B receives both, decripts the crypted message and checks for being equal with the clear one. If it is, then it must be A that sent the message, since it was encrypted with A's private key, which only A has.

Problems:

  • Asymmetric ecryption is slow for long messages. Solution: instaed of sending clear msg and E(msg, kprivate), we send h(msg) and E(h(msg), kprivate), that is we apply what's above to the digest of the message instaed of the message itself.
  • Sign means to sign the code (the binary code that represents a file), however many files have dynamic content (es: depending on date), thus the content of it can change making signatures useless. Solution: only sign types of document that can't change

Authenticity + confidentiality: matrioska.

  1. A encrypts with its private key (Auth.), then encrypts with public key of B (confid). Send message
  2. B decrypts with its private key (conf), then decrypts with As public key (auth)
    RSA standard also provides confidentiality
