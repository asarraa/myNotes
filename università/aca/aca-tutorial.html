Complex pipeline

  • If the Issue stage has an unlimited buffer, there can be multiple instructions stalling in the issue stage at the same clock cycle, however the actual Issuing should be done one instruction at the time, while the others are still stalling. In general, if some instruction are stalling in the issue stage, the next instruction can enter the issue stage (without being marked as stalling), but cannot go on since in the next stages it will be stalled behind the previous instructions, it cannot happen that instruction B starts execution before instruction A (if order is A,B): it's the in order issue property!
    By removing the in order issue property, I3 can start as soon as possible:
  • if an instruction should enter a certain stage (es: Decode) but cannot enter the next stage due to rules (es: cannot enter Issue since there are some WAR), then the instruction can enter the first stage (D) but its signed as Stalled, since it can be completed, then we will need a clock cycle of completion (D without any stall sign), and the next CC it can go to the next stage (Issue). It will be something like Ds Ds Ds D I. While different is the case in which the stage has completed but we realize we cannot go to the next stage for some problems with other instructions (es: other instructions are in the W stage and we are in E stage, in this case we would have something like E Es Es W) --> see EX5-s62
  • If an instruction is in stall in the fetch stage, then the next instruction is postponed until it exits from F stage , since we need in order decode-> see EX5-s63
  • Structural hazards (FU busy) cause stall in the Issue stage, since instruction cannot start Execution.
  • WB and Issue of two RAW dependent instruction can happen in the same CC
  • WB and Issue of two WAW-WAR dependent instructions can happen in the same CC
  • Branch: if there is not branch prediction, we need to stall before branch instruction reaches the stage in which the branch is resolved (ex ISSUE stage) and then wait the branch resolution in order to fetch the instruction after the branch. In case there is branch prediction, then we can proceed with fetching and issuing the next instruction after the branch, before the branch was solved. In this cases, as soon as the branch was solved, we need to ask if flushing the predicted instruction is needed. --> EX5-s70

Cases:

  • Instruction above is in stall in a stage S, the instruction below can only go on if the stalled instruction is not occupying its stage
    • If we need to fetch and S is fetch, we postpone the fetch until the stall terminate
    • If we are in D and should go in I, generally we can perform the D stage but will stall there because we cannot go in I, but in case the I has a buffer, we can go and stall there. Ex3:
    • Otherwise, we stall.
      Remember: each stage always need to spend a CC as non stalled, before or after the stalls.
  • In case of WAR-WAW. We stall in D until our D (non-stalled any stall) is in the previous column wrt the WB of instruction we depend on.
  • In case of RAW, we stall in I until our I (non-stalled) is in the same column as the WB of instruction we depend on.
  • In case on struct hazard on WB, we stall on E until we can perform our WB.

Case of branch: we can start fetching the next instruction as soon as the target is computed (depending on the assumption, it can happen in the fetch stage, so we will insert a new instruction which is stalling in F until the branch resolution), however the instruction will be stalled until the branch is resolved (depending on the assumption, it can happen at ID stage, after that, the actual F can happen). In case of branch prediction, we can avoid the stalling in fetch but we need to ask ourselves if we need to flash after the branch resolution. (ex5)

VLIW

  • In VLIW schema we insert instructions in the clock cycle they enter the IF unit. Fetch and decode happen simultaneously for operations in the same instruction. This has the following consequences:
    • Operations in the same instruction will enter the execution unit after having their operands read and stored in "private" registers. This means that reads and writes to the same register can cohesist, since they will operate on different copies of the same value--> WAR and WAW safe for instruction in the same bundle.
    • In case we have early evaluation of branch, we need understand when the operators will be available for the branch to execute, in particular if the branch has early evaluation (in ID stage). For example, here the bne has been positioned after seing that the register s4 was available 1 CC later.
  • When representing VLIW scheduling, we use IF, ID, EX, WB as in the example above
  • VLIW code must be conflict-free means that we need to schedule the code in such a way that the dependencies don't cause hazards.
  • If pipeline units are described as non-pipelined, we need to wait that an instruction is completed (that is, the clock cycles are needed are over) before putting a new one in the same functional unit.
  • Can appear with different scheduling algorithms:
    • ASAP: instructions are scheduled as soon as possible
    • In order issue: instructions are scheduled so to respect the order, that is if operation B appears after operation A, then B cannot be scheduled after A, but at most in the same instruction
  • If there is no assumption at all about branch, we can consider it a traditional branch with evaluation in the EX stage.
  • If the branch is the last operation, it should be scheduled in the last long instruction

Scoreboard

  • flag in scoreboard: means "ready" but also "don't touch it is mine" set to yes when instruction is issued and is in read operand (that is, the register is ready for me to use it and we need to tell everybody that I need to use that register, so they don't touch it), is set to no when instruction is in Exec since it is being used (I need everybody to know that I'm using it and they can touch it).
    07.ACA25-scoreboard.pdf > page=42&annotation=2432R: How to set Rj and Rk? We check the respective register in the register result status, if the cell is occupied by an operation that means that there is a FU which will need to write that register, and so the respective Rj (Rk) will be set to No, otherwise if the cell is empty we set it to Yes

  • Trick for scoreboard table Remember, scoreboard is in order issue. If we see out of order issue, we already know it is wrong

  • Only one WB per CC

    • Since in case of WAR it is not possible to perform WB, the results of instructions are held in FU until it is possible to write back, for this reason, if I have instruction A that completes execution in CC 5 and WB in CC 6, and instruction B that starts execution in CC 5, it must have used a different FU wrt the one used by instruction A, in fact the FU will be occupied in holding the result of A until CC 6

  • Stalling in scoreboard Scoreboard stalls in Issue if spots WAW or structural hazards, while stalls in Read operand if it spots RAW hazard, stalls in WB if it spots WAR hazards (this, until the instruction from which it is dependend has completed the read operand phase). In all these cases, we write in our table the CC in which the stall ends. For example, if instruction B is dependend from A with a WAW dependency, then B will not have its ISSUE numbered untio instruction A performs its WB.

  • Exe start is always read operands +1. Exe complete is computed counting latency from start exe, wb is one CC later than exe complete, unless war or struc hazards (es, with 3 CC Latency: read operands 4, start exe 5, complete exe 7, wb 8)

Scoreboard con Explicit register renaming

All D instruction occupy 64bits, depending on the size of register you occpy one or more (in general they occupy 2 register of 32 bits).
In the Issue stage we no more check for WAW, instaed we allocate a new physical register for the destination register, that is, we update the mapping in the rename table, based on the free list (the list of free physical registers).

  • WAW and WAR are solved by design
  • Notice that if two different instructions write the same register, the virtual-p,,hysical mapping of this register is changed twice, however this is not a problem even if the first instruction has not completed the write back yet. Even if the mapping in the rename table is different from what it is expecting, it will write in the phyisical register it allocated at issue stage, and it will still result busy (so will not appear in the free list), eve if it does not appear in the rename table

Tomasulo

  • We reserve a functional unit for the execution of an instruction only at START EXE stage
  • We have to wait one clock cycle after a write back for a depending instruction to start (as in scoreboard)
  • Only one FU performing WB at a time (since we only have one CDB), in case multiple instruction need to write back, we can choose arbitrarily which ones goes first (or follow the directive of the exercise )
  • Reservation station are marked as free only after the WB!
  • Loads vs stores: Loads can be freerely moved among themselves
  • Loads vs stores, stores vs stores When a load address is computed (this happens in the address unit that is before issuing the instruction in a reservation station, so in the exercises we assume the address computation is done in the issue stage. N.B. Address computation is performed in order) it's compared with addresses in active store buffers (memory addresses targeted by stores). If there's a match, the load must wait for the conflicting store to complete (cannot be executed before it), otherwise we can move start executing the load before the store (for example if the store is waiting for a raw). Similarly, stores check for conflicts with other memory operations.

Tomasulo with Rob

  • The Write is now the stage in which we are going to write the result to the ROB, while the Commit is the stage in which we write the result to the register file. Later instructions can start execution if their operands registers are available in the ROB (even if their are not yet in the register file).
  • If the ROB Is full, we get a structural hazard, this means we cannot issue new instructions.
  • ROB allows for speculative execution, this executing instructions before we know if we should (es: second iteration of a cycle), however we need to keep a counter that counts the CC elapsed to the misprediction, it starts from the first instruction that we don't know if we should execute: when we reach the CC number that we needed to know that we mispredicted, we need to flush the mispredicted instructions from the ROB

Cache penalties

Cache penalties are paid in the execute stage.
The processor doesn't have any knowledge about memory. Read operand phase access register file, not memory. Execution phase of a memory operation means: compute the address (where we should go in the memory) and then ask for the value (if its a load) or send the value (is it's a store), it's here that we pay the cache penalties.

Cache coherency

  • If a processor has a resouce invalid, it will stay so even if another process reads the new resource (it will not become shared)
  • if a processor has a resource modified, as soon as another process reads it , it will become shared for both
