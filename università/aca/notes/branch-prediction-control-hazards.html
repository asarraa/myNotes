Branch prediction - control hazards

source

Introduction

Control hazards can cause a greater performance loss for a MIPS/RISC pipeline than do data hazards. When a branch is executed, it may or may not change the PC to something other than its current value plus 4. Recall that if a branch changes the PC to its target address, it is a taken branch; if it falls through, it is not taken, or untaken. If instruction i is a taken branch, then the PC is usually not changed until the end of ID, after the completion of the address calculation and comparison. In fact, remember that one optimisation from the simple pipeline is moving the breanch condition check from the EX stage to the ID stage, however, even with this optimisation, there still is a one cycle penalty. Here we try to reduce this penalty.

Simple solution: pipeline freeze/flush

Simplest method of dealing with branches: redo the fetch of the instruction following the branch instruction in order to "wait" for the result of the ID stage of the previous instruction (and in order to understand which is the right instruction to fetch). We notice that if the branch is untaken, then the repetition of the IF stage is unnecessary because the correct instruction was indeed fetched.
Of course, the branch penalty is fixed and cannot be reduced by software.

Branch prediction techniques

Main goal: try to predict as early as possible the outcome of a branch instruction.

The performance of a branch prediction depends on:

  • Accuracy, measured in terms of percentage of incorrect predictions.
  • Cost (or misprediction penalty) of an incorrect prediction: measured in terms of time lost to execute useless instructions given by the processor architecture. Notice that the cost increases for deep pipelined processors (more stages to flush).
  • Branch frequency: the importance of accurate branch prediction is higher in programs with higher branch frequency.

Static Branch Prediction Techniques

The actions for a branch are fixed for each branch during the entire execution. The actions are fixed at compile time.

Branch always not taken (Predicted-Not-Taken)

We assume the branch will not be taken, thus the sequential instruction flow we have fetched can continue as if the branch condition was not satisfied. Then:

  • If the prediction is correct (branch condition not satisfied) we can preserve the performance
  • If the prediction is incorrect (branch condition satisfied), then we need to flush the next instruction already fetched (which is turned into a nop) and we restart the execution by fetching the instruction at the branch target address

Penalty: One-cycle when the prediction is incorrect.

Branch always taken

We assume the branch is always taken, thus as soon as the branch is decoded and the branch target is computed, we begin fetching and executing the instruction at the target address.

However, in the MIPS pipeline, we don't know the branch target address earlier than the branch outcome, so this approach has no advantage for this particular pipeline.

Backward taken forward not taken (BTFNT)

The prediction is based on the branch direction:

  • Backward-going branches are predicted as taken
    Example: the branches at the end of loops go back at the beginning of the next loop iteration
  • Forward-going branches are predicted as not taken

Profile-driven prediction

The branch prediction is based on profiling information collected from earlier runs. The method can use compiler hints

Delayed branch technique

The compiler statically schedules an independent instruction in the branch delay slot (that is, the slot for the next instruction on which we are not sure is exploited with an independent instruction).
If we assume a branch delay of one cycle (as for MIPS), we have only one delay slot.

There are three ways in which the branch delay slot can be scheduled:

  • From before: The branch delay slot is scheduled with an independent instruction from before the branch.
  • From target: the branch delay slot is scheduled from the target of the branch.
    Preferred when the branch is taken with high probability (such as backward branches)
  • From fall-through: The branch delay slot is scheduled from the not-taken fall-through path. That is, we follow the path of instructions as if the branch was not taken and select an independent instruction.
    Preferred when the branch is not taken with high probability (such as forward branches)

Compilers must fill the slots

The MIPS compiler always tries to schedule a branch-independent instruction after the branch. However, usually compilers are able to fill about 50% of slots with useful instruction, while the remaining slots are filled with nops. In deep pipelined processor there are more slots to fill, thus is more challenging to fill all the slots with useful instructions.

Canceling or nullifying branch

To improve the ability of the compiler to fill the branch delay slot most processors have introduced this type of branch prediction technique.

In the canceling (nullifying) branch mechanism, the compiler encodes prediction information directly into the branch instruction, specifying whether the branch is expected to be taken or not taken. Based on this prediction, the CPU fetches the next instruction accordingly. If the prediction suggests that the branch will be taken, the CPU immediately fetches the instruction located at the branch target address. If the prediction indicates that the branch will not be taken, the CPU instead fetches the next sequential instruction by incrementing the program counter by four.

After fetching, the CPU evaluates the actual branch condition to determine whether the prediction was correct. If the prediction was correct, execution proceeds normally. However, if the prediction was incorrect, the instruction in the branch delay slot is nullified, (becomes nop) rather than being executed.

Canceling vs branch always taken

The key distinction between these two mechanisms lies in the instruction fetch behavior. In the "branch always not taken" mechanism, the CPU always fetches the next sequential instruction before checking the actual branch condition, leading to a pipeline flush when the prediction is incorrect. In contrast, the canceling (nullifying) branch mechanism fetches the instruction indicated by the embedded prediction, reducing the likelihood of pipeline flushes and improving execution efficiency by selectively nullifying the delay slot instruction only when a misprediction occurs.

Dynamic Branch Prediction Techniques

Basic Idea: To use the past branch behavior to predict the future.

Dynamic branch prediction is based on two interacting mechanisms:

  • Branch Outcome Predictor (BOP): To predict the direction of a branch.
  • Branch Target Predictor (BTP): To predict the branch target address in the case of taken branch.
    The Instruction Fetch Unit uses these modules to predict the next instruction to read in the I-cache:
    If the branch is not taken -> PC is incremented
    If the branch is taken -> BTP gives the target address

Branch Target Buffer (BTB)

Is a cache storing the predicted branch target address for the next instruction after a branch.

In the IF stage, we access the BTB using the instruction address of the fetched instruction (which is a branch) to index the cache.

The BTB is composed by entries with the following structure:

Branch History Table (or Branch Prediction Buffer)

is a small memory indexed by the lower portion of the address of the branch instruction. The memory contains one or more bits that say whether the branch was recently taken or not. This buffer is effectively a cache where every access is a hit.

It is important to notice that prediction is just an hint that is assumed to be correct. If the hint turns out to be wrong (ex: prediction bit could have been put there by another branch with the same low-portion address bits), the prediction bit is inverted and stored back. The pipeline is flushed, and the correct sequence is executed.

Accuracy of the BHT
A misprediction can happen in two different cases:

  • The prediction is incorrect for that branch,
  • The same index has been referenced by two different branches (with the same lower portion of the address), and the previous history refers to the other branch
    For this reason, it is convenient to increase the number of rows in the BHT or to use a hashing function.
1-bit Branch History Table

The simplest type of BHT is the one which encodes the prediction in a single bit:

  • 1: branch taken
  • 0: branch not taken
2-bits Branch History Table

In a 1-bit branch history table, a single misprediction causes the prediction bit to be flipped, causing a loss in prediction accuracy. For this reason we would prefer the prediction to be wrong twice before the prediction bit its changed.
For each index in the table, 2 bits are used to encode the four states of a finite state machine.

For example, in a loop branch, at the last loop iteration, we do not need to change the prediction.

  • Otherwise, it is predicted as untaken.

As in the 2-bit scheme, the counter is incremented on a taken branch and decremented on an untaken branch.

Correlating branch predictors (or 2-level Predictors)

BHT predictors use only the recent behavior of a single branch to predict the behavior of that branch, however the recent behavior of other branches can influence the prediction of the current branch. Correlating branch predictors exploit these information.

(1,1) Correlating branch predictors

(1,1) Correlating Predictors means a 1-bit predictor with 1-bit of correlation (or global history): the behaviour of the last branch is used to choose among a pair () of 1-bit branch predictors.

(2,2) Correlating Branch Predictors

(2,2) Correlating Predictors means a 2-bit predictor with 2-bit of correlation: the behavior of the last branch is used to choose among four () of 2-bit branch predictors.

In this particular case, each BHT entry contains a 4-bit low-portion address.

2-bit vs (2,2) correlating: performance

A 2-bit predictor with no global history is simply a (0,2) correlating predictor. The (2,2) outperforms the simple 2-bit predictor both in case of same number of entries in the BHTs and when the simple one has unlimited number of entries.

(m, n) Correlating Branch Predictors

Generally, correlating predictor records the last branches outcomes to choose from different BHTs, each of which is an n-bit predictor.

2-level Adaptive Branch Predictors

A 2-Level Adaptive Branch Predictor works by using two levels of history to make predictions:

  1. First-level history: This is stored in a Branch History Register (BHR), which tracks the outcomes (taken or not taken) of the most recent branches. The size of the BHR is determined by k, the number of recent branches it records.
  2. Second-level history: This is stored in a Pattern History Table (PHT), which contains several 2-bit saturating counters. Each counter reflects how likely a branch is to be taken or not taken based on past behavior.

The BHR is used to index the PHT and select the appropriate 2-bit counter. The prediction is then made using the value of the selected counter, following the same process as in the basic 2-bit counter prediction scheme. This allows the predictor to adapt more accurately to different branching patterns.
