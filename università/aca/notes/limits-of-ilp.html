The lecture explores the concept of Instruction-Level Parallelism (ILP), which refers to the degree to which instructions of a program can be executed simultaneously. In an ideal world, if all dependencies between instructions could be resolved and resources were unlimited, multiple instructions could be issued, executed, and completed in the same cycle, significantly enhancing processor performance.

To understand ILP, the lecture begins with its foundational definition: ILP is the potential overlap in the execution of instructions that are independent of each other. However, such overlap is possible only when there are no structural hazards (i.e., competition for hardware resources), no data hazards (RAW, WAR, WAW), and no control hazards (like branch mispredictions). In realistic conditions, many of these issues arise and must be handled effectively to take advantage of ILP.

To improve ILP, modern processors have adopted superscalar architectures. Superscalar processors are capable of issuing multiple instructions per clock cycle. This enhancement requires sophisticated mechanisms: instruction fetching at higher bandwidths, dynamic scheduling to manage instruction dependencies at runtime, and advanced branch prediction techniques to minimize the disruptions caused by control dependencies. The lecture emphasizes that moving beyond a single-issue, in-order execution model is necessary to exploit more ILP. Superscalar design is a significant step in that direction.

The heart of the discussion revolves around the limits of ILP. To analyze these limits, the lecture introduces an idealized machine model — a theoretical CPU with perfect capabilities: infinite register renaming (eliminating WAW and WAR hazards), perfect branch and jump prediction, flawless memory alias analysis (so stores and loads can be reordered freely when safe), and all instruction latencies reduced to one cycle. This model can also issue and complete an unlimited number of instructions per cycle.

Yet, even with these perfect conditions, simulations using SPEC92 benchmark programs (a set of real-world floating point and integer-heavy applications) show that the upper bound on instruction-level parallelism is limited. For example, in some cases the ideal ILP measured is 50 to 150 instructions per cycle, depending on the program. But these are theoretical ceilings. Real-world implementations fall well below due to physical and practical constraints.

One such constraint is the issue window size — the set of instructions the CPU can examine in order to choose which to execute in parallel. Ideally, a processor would look far ahead in the instruction stream to find independent instructions, but this quickly becomes computationally expensive. For instance, a window of 2000 instructions might require millions of comparisons to detect dependencies, which is infeasible in hardware. Current processors compromise with window sizes in the range of 32 to 200, which already demand thousands of comparisons per clock cycle.

Another limiting factor is the issue width, or the number of instructions that can actually be issued per clock cycle. Though a processor might analyze hundreds of instructions in a large window, it is still limited by the number of functional units, ports, and buses available. Modern processors rarely go beyond 4 to 6 issue widths because increasing this number increases design complexity, power consumption, and lowers the clock frequency due to slower decision-making logic.

A compelling example used in the lecture is Intel’s P6 microarchitecture (e.g., Pentium Pro, Pentium II/III), which introduced dynamic scheduling with out-of-order execution. The P6 fetches and decodes up to three IA-32 instructions per clock cycle, converts them into simpler micro-operations (uops), and places them into an instruction pool. These uops are then scheduled and executed out-of-order, depending on operand availability and resource readiness. A 14-stage pipeline handles in-order fetch and decode, out-of-order execution, and in-order commit. Even in this fairly advanced architecture, only a limited number of instructions (typically 3 to 5) can be retired per cycle.

The lecture also touches upon the RISC vs. CISC debate. RISC (Reduced Instruction Set Computing) architectures use simple, fixed-length instructions, facilitating easier ILP extraction and pipeline design (examples include ARM, MIPS, RISC-V). CISC (Complex Instruction Set Computing), like x86, uses more complex instructions that do more per instruction but can be harder to pipeline. However, modern CISC processors like Intel’s translate complex instructions into RISC-like micro-ops internally, effectively behaving like a RISC engine under the hood.

In exploring ILP further, the lecture addresses energy efficiency and scaling issues. Many techniques that increase ILP also increase power consumption disproportionately. For example, wider issue widths require additional logic that scales faster than the performance gain. The result is a growing gap between theoretical peak performance and sustained real-world throughput. This efficiency issue is one reason ILP improvements have plateaued in recent years.

As a result, processor designers have shifted focus. Since 2002, single-core performance improvements have slowed dramatically (down from ~55% per year to ~20% per year), as the limits of ILP and frequency scaling have become apparent. The solution has been to move toward explicit parallelism, where the burden of parallel execution is shared with the programmer or compiler, rather than relying solely on hardware. This includes strategies like multi-core architectures, SIMD (Single Instruction, Multiple Data) instructions, and sub-word parallelism.

In closing, the lecture argues that while ILP has brought significant performance gains — especially between 1985 and 2002 — it faces inherent limits. These limits are dictated by both fundamental program structure and practical hardware constraints. Moving forward, extracting performance will increasingly rely on broader forms of parallelism and improved energy efficiency rather than simply increasing ILP.
