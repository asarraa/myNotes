Introduction and Context

Tomasulo's algorithm is a hardware-based dynamic scheduling technique developed by Robert Tomasulo at IBM in 1967 for the IBM 360/91, just three years after CDC's scoreboard approach. Like scoreboarding, it addresses the challenge of executing instructions efficiently in the presence of data dependencies, but with several significant advances.

The fundamental goal remains consistent: achieve high performance without requiring special optimizing compilers. This approach has influenced numerous modern processor designs, including the Alpha 21264, HP 8000, MIPS 10000, Pentium II, and PowerPC 604.

Distributed Control: A Fundamental Shift from Scoreboarding

The most critical conceptual difference between Tomasulo's algorithm and scoreboarding lies in the control architecture. While the scoreboard used a centralized mechanism to track and manage all dependencies and instruction execution, Tomasulo's approach distributes this control throughout the processor.

In the scoreboard architecture, a single central unit maintained all information about instruction status, functional unit status, and register status. This centralization created a potential bottleneck and limited the scalability of the design. Tomasulo's algorithm breaks away from this model by distributing control logic directly to the functional units themselves.

This distributed approach is implemented through specialized buffers called "Reservation Stations" that are associated with each functional unit. When an instruction is issued to a particular functional unit, it becomes an entry in the corresponding reservation station. This distributed architecture allows for more parallel decision-making and potentially greater scalability.

Core Components of Tomasulo's Architecture

Reservation Stations

Reservation stations are the heart of Tomasulo's algorithm. Each functional unit has one or more reservation stations associated with it. These stations serve as buffers that hold instructions waiting to execute, along with their operands or information about where those operands will come from.

Each reservation station contains several critical fields:

  • A tag that uniquely identifies the reservation station
  • The operation code indicating what operation should be performed
  • Fields for source operand values (Vj, Vk) if they're already available
  • Pointers to other reservation stations (Qj, Qk) that will produce source operands if they're not yet available
  • A busy flag indicating whether the station contains an active instruction

The key insight here is that for each operand, either the value is available (stored in V) or the system knows exactly which reservation station will produce it (tracked in Q). This mechanism enables precise tracking of dependencies in a distributed fashion.

Register File

The register file in Tomasulo's architecture is enhanced with additional tracking fields. Each register contains not just a value but also a field indicating which reservation station (if any) will produce the next value for that register. Specifically:

  • A value field (V) containing the actual data
  • A pointer field (Q) identifying the reservation station that will write to this register

If the Q field is zero, it means no instruction is currently scheduled to write to this register, and the value in V is valid. If Q is non-zero, it points to the reservation station that will eventually produce the register's value.

Common Data Bus

A critical component that enables Tomasulo's distributed architecture is the Common Data Bus (CDB). This bus connects the outputs of all functional units to all reservation stations and the register file. When a functional unit completes an operation, it broadcasts both the result value and its source identifier on the CDB.

This broadcast mechanism serves a crucial role in handling data dependencies. Each reservation station waiting for an operand watches the CDB for results. When a result appears whose source identifier matches what the reservation station is waiting for, it captures that value directly. This direct propagation of results to all potential consumers eliminates the need for a centralized controller to manage these dependencies.

Load/Store Units (store and load buffers)

Load and store operations are handled as specialized functional units with their own buffers:

  • Load buffers track memory addresses to be read
  • Store buffers track both addresses and values (or pointers to where values will come from)

These buffers allow memory operations to participate in the same out-of-order execution paradigm as computational instructions, with special handling to preserve memory consistency.

Register Renaming: Eliminating Name Dependencies

One of the most powerful innovations in Tomasulo's algorithm is its implicit register renaming mechanism. Traditional architectures use a fixed set of registers identified by names (R1, R2, etc.). This naming creates artificial dependencies between instructions that happen to use the same register names but don't actually share data.

Tomasulo's algorithm breaks these artificial dependencies by decoupling architectural registers (those visible to the programmer) from physical storage locations. When an instruction enters a reservation station, its register operands are "renamed" by replacing them with either actual values (if available) or pointers to the reservation stations that will produce those values.

This renaming automatically eliminates WAR and WAW hazards, which are not true data dependencies but merely name dependencies:

  • WAR (Write After Read) hazards occur when an instruction wants to write to a register that a previous instruction still needs to read. With renaming, the earlier instruction already has either the value it needs or a pointer to where it will come from, so the later write doesn't interfere.
  • WAW (Write After Write) hazards occur when multiple instructions want to write to the same register. With renaming, each write effectively goes to a different physical location (reservation station), eliminating interference.

By resolving these name dependencies, Tomasulo's algorithm allows instructions to execute in parallel that would otherwise have to wait due to register name conflicts. The reservation stations effectively provide more registers than are visible in the architecture, enabling optimizations beyond what compilers can achieve with the limited architectural register set.

The Three Stages of Execution

Tomasulo's algorithm divides instruction execution into three key stages: Issue, Execution, and Write Result.

1 - Issue Stage

During the Issue stage, instructions are taken from the instruction queue and assigned to reservation stations if available. This stage is performed in program order, similar to the Issue stage in scoreboarding. However, the handling of dependencies is quite different.

For each instruction:

  1. If all reservation stations are full for the required functional unit, the instruction stalls (structural hazard). Otherwise, the instruction is placed in an available reservation station.
  2. For each source register operand:
    • If the value is available in the register file (Q=0), the value is copied directly to the reservation station.
    • If the value is being computed (Qâ 0), the reservation station records the Q value as a pointer to where the result will come from.
  3. The register file's Q field for the destination register is updated to point to this reservation station.

Unlike scoreboarding, Tomasulo's algorithm doesn't need to check for WAW hazards during issue, they're automatically handled through register renaming. The instruction can be issued as long as a reservation station is available, regardless of register dependencies.

2 - Execution Stage

During the Execution stage, each reservation station operates independently, monitoring its operands and beginning execution when they're all available. By delaying execution until operands are available, RAW hazards are avoided. Notice that several instructions could become ready in the same clock cycle for the same FU

A reservation station is ready to execute when:

  • Its busy flag is set (it contains an instruction)
  • Both operands are available (=0 and =0)

If operands aren't ready, the reservation station watches the Common Data Bus for results. When a result appears in the CDB and its source tag matches what the reservation station is waiting for, it captures that value and clears the corresponding Q field.

For load/store operations, execution happens in two steps, for preserving memory consistency.

This stage enables true out-of-order execution, as instructions execute based solely on data availability, not program order.

3 - Write Result Stage

During the Write Result stage, completed results are distributed to where they're needed via the Common Data Bus.

When a functional unit completes an operation:

  1. It broadcasts the result along with its reservation station identifier on the CDB.
  2. All watching components receive this broadcast:
    - The register file updates the appropriate register if its Q field matches the broadcasting station.
    • Reservation stations capture the result if their Qj or Qk fields match.
  3. Store operations write data to memory during this stage.
  4. The reservation station is marked as available for new instructions.

This distributed notification system is what allows Tomasulo's algorithm to efficiently handle dependencies without a centralized controller. All potential consumers of a result monitor the CDB and capture values when they appear, eliminating the need for complex dependency tracking structures.

Special Handling for Memory Operations

Memory operations require special care to maintain program correctness while allowing out-of-order execution. Loads and stores go through a two-step process:

First, they compute their effective addresses in the address unit. Then, in the memory unit :

  • Loads access memory and broadcast the loaded value on the CDB
  • Stores wait for their data value to be available, then write to memory

The challenge comes when multiple memory operations might access the same location. Loads and stores can execute out of order if they access different memory locations, but reordering operations to the same location could cause incorrect results:

  • Reordering a store before a load creates a RAW hazard
  • Reordering a load before a store creates a WAR hazard
  • Reordering two stores creates a WAW hazard

This because we are talking about actual registers, for memory operations the decoupling among registers and values doesn't apply and so we can get hazards.
To handle this, Tomasulo's implementation performs memory disambiguation. When a load address is computed, it's compared with addresses in active store buffers. If there's a match, the load must wait for the conflicting store to complete. Similarly, stores check for conflicts with other memory operations.

This dynamic disambiguation requires significant hardware resources, as each buffer needs fast associative lookup capabilities. It represents an alternative to static disambiguation performed by compilers.

Tomasulo's Algorithm with Hardware Speculation (ROB)

This variant of Tomasulo's algorithm extends the original mechanism to support hardware speculation (es: speculation about branch predictions). The key component added is the Reorder Buffer (ROB). The ROB functions as a buffer that maintains the result of an instruction after its execution is completed, but before the instruction is committed. The crucial function of the ROB is to ensure that instructions commitâmeaning their results definitively modify the visible processor state (such as architectural registers or memory)âin the original program order.

This in-order commitment is fundamental for supporting precise exceptions: an exception is handled only when the instruction that caused it reaches the head of the ROB, ensuring that all previous instructions have completed and the state is consistent with sequential execution up to that point.

The ROB also integrates the function of the store buffer from the original Tomasulo algorithm, ensuring that memory updates occur only when the store instruction commits. Additionally, the ROB can provide operands to subsequent instructions via the common data bus, serving as a temporary source for results pending commitment.

Advantages and Limitations

Tomasulo's algorithm provides several significant advantages over scoreboarding:

It eliminates WAR and WAW hazards through register renaming, removing artificial dependencies that would otherwise limit parallelism. The distributed control logic allows for more scalable designs and potentially higher performance. The Common Data Bus provides an efficient mechanism for result forwarding, ensuring operands get to where they're needed without centralized coordination.

However, these advantages come with some costs. The hardware implementation is more complex than scoreboarding, requiring reservation stations for each functional unit and the Common Data Bus infrastructure. Memory disambiguation adds additional complexity, especially for processors supporting many in-flight memory operations.

The single Common Data Bus can also become a bottleneck in designs with many functional units, as only one result can be broadcast per cycle. More advanced implementations often use multiple buses to alleviate this limitation.

Conclusion

Tomasulo's algorithm represents a significant advancement in dynamic scheduling techniques. By distributing control through reservation stations, implementing implicit register renaming, and using the Common Data Bus for result distribution, it achieves greater parallelism than earlier approaches like scoreboarding.

The core innovation lies in decoupling architectural register names from physical storage locations, which eliminates artificial dependencies and allows more instructions to execute in parallel. This approach laid the groundwork for many modern processor designs and continues to influence CPU architecture today.

Understanding Tomasulo's algorithm provides crucial insights into how modern processors achieve high performance through aggressive out-of-order execution while maintaining program correctness.
