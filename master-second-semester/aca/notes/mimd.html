MIMD

Introduction to MIMD Architectures

Multiple Instruction, Multiple Data (MIMD) architectures represent a fundamental approach to parallel computing that offers flexibility and scalability for modern computational needs. Unlike simpler parallel architectures, MIMD systems can operate as single-user machines delivering high performance for individual applications, as multiprogrammed multiprocessors handling numerous tasks simultaneously, or as a combination of both functions. A significant advantage of MIMD systems is that they can be constructed using standard CPUs, which is the common approach for nearly all contemporary multiprocessors.

The core of MIMD functionality lies in thread-level parallelism. To effectively utilize a MIMD system with n processors, you need at least n threads or processes to execute. These independent threads are typically identified by the programmer or created by the compiler. A thread can range from being a large, independent process to parallel iterations within a loop. The crucial distinction between MIMD systems and other processors like superscalar CPUs is that in MIMD, parallelism is identified by software rather than hardware.

Memory Organization in MIMD Systems

The way parallel processors share data is fundamental to understanding MIMD architectures. There are two primary memory address space models:

Shared Memory Architectures

In shared memory architectures, there exists a single logically shared address space where any processor can make memory references to any memory location. The address space is shared among processors, meaning the same physical address on two different processors refers to the same location in memory. The processors communicate through shared variables in memory, with implicit management of the communication through load/store operations. This is the oldest and most popular model for parallel processing.

It's important to note that having a shared memory does not necessarily mean there is a single centralized memory. The concept of sharing refers to the logical address space, not the physical memory organization.

Message Passing Architectures

In contrast, message passing architectures utilize multiple and private address spaces. The address space is logically disjoint and cannot be addressed by different processorsâthe same physical address on two processors refers to two different locations in two different memories. The processors communicate by sending and receiving messages, requiring explicit management of communication through send/receive primitives. One advantage of this approach is that it eliminates cache coherency problems among processors.

Physical Memory Organization

The physical organization (which is orthogonal with respect to the logical one) of memory in MIMD systems can be categorized into:

Centralized Shared-Memory Architectures

These typically support at most a few dozen processor chips (less than 100 cores) and feature large caches with a single memory divided into multiple banks. These are often called Symmetric Multiprocessors (SMP) and follow the Uniform Memory Access (UMA) style of architecture, where all processors have equal access time to all memory locations.

Distributed Memory Architectures

These architectures are designed to support large processor counts and require high-bandwidth interconnects. While they allow for greater scalability, they introduce the challenge of data communication among processors. These systems follow the Non-Uniform Memory Access (NUMA) model, where memory access time depends on the memory location relative to the processor.

It's crucial to understand that the concepts of addressing space (single or multiple) and physical memory organization (centralized or distributed) are orthogonal. This means that multiprocessor systems can have a single addressing space combined with distributed physical memory.

Programming Models

Shared Memory Programming

In the shared memory model, a program is a collection of threads of control that can be created dynamically during execution in some programming languages. Each thread has a set of private variables (like local stack variables) and shared variables (such as static variables, shared common blocks, or global heap). Threads communicate implicitly by writing and reading shared variables and coordinate by synchronizing on these shared variables.

Message Passing Programming

In the message passing model, a program consists of a collection of named processes, usually fixed at program startup time. Each process has its own thread of control and local address space, with no shared data. Logically shared data is partitioned over local processes. Processes communicate through explicit send/receive pairs, with coordination implicit in every communication event. The Message passing interface (MPI) is the most commonly used software framework for this model.

Comparing Shared Memory and Message Passing

Both models have their strengths and weaknesses, and the choice between them depends on the specific requirements of the program.

Advantages of Shared Memory:

  • Implicit communication through loads/stores
  • Low overhead when data is cached

Disadvantages of Shared Memory:

  • Complex to build in a way that scales well
  • Requires synchronization operations
  • Difficult to control data placement within the caching system

Advantages of Message Passing:

  • Explicit communication through sending/receiving messages
  • Easier to control data placement (no automatic caching)

Disadvantages of Message Passing:

  • Message passing overhead can be quite high
  • More complex to program
  • Introduces questions about reception technique (interrupts/polling)

Specific Challenges of Message Passing in Massively Parallel Processors:

  • All data layout must be handled by software
  • High software overhead for message passing
  • Early machines required OS intervention for each message (taking 100s-1ms per message)
  • Even with user-level access to the network interface, there is still dozens of cycles of overhead
  • While sending messages can be relatively inexpensive (similar to store operations), receiving messages is costly and may require polling or interrupts

Bus-Based Symmetric Shared Memory

These systems continue to dominate the server market and serve as building blocks for larger systems, even making their way to desktop computers. They are attractive both as throughput servers and for parallel programs due to:

  • Fine-grain resource sharing
  • Uniform access via loads/stores
  • Automatic data movement and coherent replication in caches
  • Cost-effective and powerful extension of uniprocessor systems

They use normal uniprocessor mechanisms to access data, with the key innovation being the extension of the memory hierarchy to support multiple processors.

The Challenge of Cache Coherence

In shared-memory architectures, both private data (used by a single processor) and shared data (used by multiple processors for communication) are cached. When shared data is cached, the shared value may be replicated in multiple caches. While this replication reduces access latency and required memory bandwidth, it also reduces contention for shared data read by multiple processors simultaneously.

However, private processor caches introduce a significant problem: copies of a variable can exist in multiple caches, and a write by one processor may not be immediately visible to others. This creates the challenge of cache coherence.

The Cache Coherence Problem Illustrated

Consider a scenario with multiple processors that have write-through caches:

  1. Memory contains a value (u:5)
  2. Processor P1 reads this value into its cache
  3. Processor P2 also reads this value into its cache
  4. Processor P3 reads this value as well
  5. Processor P3 writes a new value (u=7) to its cache
  6. Processors P1 and P2 read the value again

In this scenario, processors see different values for the variable after the write event. With write-back caches, the value written back to memory depends on which cache happens to flush or write back its value. Processes accessing main memory may see very outdated values, which is unacceptable for program execution.

Defining Coherency

Coherency doesn't strictly mean that "any read must return the most recent write," as this would be too restrictive and difficult to implement. Instead, a better definition is that "any write must eventually be seen by a read" and all writes are seen in proper order (serialization).

To ensure this, two rules are followed:

  1. If a processor writes to a location and another processor reads it, the write will be seen by the reading processor if the read and write are sufficiently far apart and no other writes occur in between.
  2. Writes to a single location are serialized: two writes to the same location by any two processors are seen in the same order by all processors.

Memory Consistency

We cannot expect that a read immediately sees a write that happened just moments before. This is the problem of memory consistency, which complements coherence:

  • Coherence defines the behavior of reads and writes to the same memory location
  • Consistency defines the behavior of reads and writes with respect to accesses to other memory locations

For simplicity, we can assume that:

  • A write does not complete (and allow the next write to occur) until all processors have seen its effect
  • A processor does not change the order of any write with respect to any other memory access

Maintaining Coherent Caches

In a coherent multiprocessor, caches provide both migration and replication of shared data items:

  • Migration allows a data item to be moved to a local cache and used transparently, reducing latency for accessing remotely allocated shared data and decreasing bandwidth demands on shared memory
  • Replication creates copies of data items in local caches for simultaneously read shared data, reducing both access latency and contention

Solutions for Cache Coherence

Hardware-based solutions to maintain coherency are implemented through cache-coherence protocols. The key to implementing a cache coherent protocol in multiprocessors is tracking the status of any sharing of a data block.

There are two main classes of protocols:

  1. Snooping Protocols
  2. Directory-Based Protocols

These protocols ensure that shared data remains consistent across multiple caches, allowing efficient parallel processing in MIMD architectures.

In conclusion, MIMD architectures provide powerful frameworks for parallel computing, with various approaches to memory organization and communication. Understanding the trade-offs between shared memory and message passing, as well as the challenges of cache coherence, is essential for designing and programming efficient multiprocessor systems.
