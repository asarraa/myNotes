Q: [03 - Binary Formats] What information does a binary format hold?
A: A binary format holds information about how the file is organized on disk, how to load it in memory, whether it is an executable or library, the entry point (if executable), the machine class (e.g., x86), and sections such as data and code.

Q: [04 - ELF Binaries] What are the components of an ELF binary?
A: The components of an ELF binary include the ELF header, program headers, and section headers. The ELF header describes the high-level structure of the binary, program headers describe how the file will be loaded in memory, and section headers describe the binary as on disk.

Q: [05 - ELF Header] What information does the ELF header contain?
A: The ELF header contains information such as the file type (executable, shared library, relocatable object), the section and program headers boundaries, and where in the file to find the other parts (the program headers and section headers).

Q: [06 - Program Headers] What is the role of program headers in an ELF binary?
A: Program headers describe how the file will be loaded in memory. They divide the data into segments and map sections to segments.

Q: [07 - Section Headers] What is the role of section headers in an ELF binary?
A: Section headers describe the binary as on disk. They define sections such as .init (executable instructions that initialize the process), .text (executable instructions of the program), .bss (statically-allocated variables), and .data (initialized data).

Q: [08 - Process Creation in Linux] What are the steps involved in process creation in Linux?
A: The steps involved are: the kernel creates a virtual address space, information is loaded or mmap'ed from the exec file to the newly allocated address space, the loader and dynamic linker load the segments defined by the program headers, and the kernel sets up the stack and heap and jumps to the entry point of the program.

Q: [09 - Process Layout in Linux] What is the layout of a process in Linux memory?
A: The layout includes the ELF program segments loaded in memory within a Linux process, organized in a structured manner.

Q: [10 - Virtual vs Physical Address Space] What is the difference between virtual and physical address space?
A: Virtual address space is the address space used by the process, while physical address space is the actual memory addresses in the hardware.

Q: [11 - The Code and the Stack] What are the key registers involved in function calls and stack operations?
A: Key registers include EIP (Instruction Pointer), ESP (Stack Pointer), and EBP (Base Pointer).

Q: [12 - Recall on Registers] What are the types of registers in a CPU?
A: The types of registers include General Purpose registers (EAX, EBX, ECX), ESP (Stack Pointer), EBP (Base Pointer), Segment registers (DS, SS), Control registers, and EFLAG registers.

Q: [13 - The call Instruction] What does the CPU do when it calls a function?
A: The CPU saves the current EIP on the stack and jumps to the function's address.

Q: [14 - Function Prologue] What is the purpose of the function prologue?
A: The function prologue allocates the activation record on the stack and saves the caller's frame so it can be restored once the callee's frame is over.

Q: [15 - Function Epilogue] What is the purpose of the function epilogue?
A: The function epilogue restores the caller's frame on the stack and returns control to the original function caller.

Q: [16 - Stack Smashing] What is stack smashing and when was it first mentioned?
A: Stack smashing is a technique to change the EIP and control the control flow. It was first mentioned in 1972 in report ESD-TR-7315 and widely popularized in 1994 by aleph1.

Q: [17 - Buffer Overflow Vulnerabilities] What is a buffer overflow vulnerability?
A: A buffer overflow vulnerability occurs when a buffer is filled without size checking, allowing an attacker to overwrite adjacent memory and potentially change the EIP.

Q: [18 - Exploitation Techniques] What are some exploitation techniques for buffer overflow vulnerabilities?
A: Exploitation techniques include using environment variables, built-in existing functions, memory that can be controlled, the buffer itself, and other variables.

Q: [19 - Environment Variables] How can environment variables be used in exploitation?
A: Environment variables can be injected into a process and used to execute code, such as putting a program into an environment variable that is executed by the command line interpreter.

Q: [20 - Built-in Functions] How can built-in functions be used in exploitation?
A: Built-in functions can be jumped to, such as skipping authentication by jumping to the function where it authenticates the user.

Q: [21 - Memory Control] How can memory that can be controlled be used in exploitation?
A: Memory that can be controlled can be filled with valid executable machine code and jumped to, allowing the execution of arbitrary code.

Q: [22 - Buffer Itself] How can the buffer itself be used in exploitation?
A: The buffer itself can be used to inject code that will be executed when the buffer overflow occurs.

Q: [23 - Personalized Parameters] How can personalized parameters be used in exploitation?
A: Personalized parameters can be added to the stack, which the called function (that we decide) will use, such as calling the exec() function with our code as a parameter.

Q: [24 - Gets Function] Why was the gets function removed from the standard C library?
A: The gets function was removed because it reads from standard input without a way to check the maximum length, making it vulnerable to buffer overflow attacks.

Q: [25 - Compiler Optimizations] What is the default stack-boundary alignment used by modern compilers?
A: The default stack-boundary alignment used by modern compilers is 16 bytes (2^4) for performance reasons on certain CPUs (e.g., Pentium III/PentiumPro (SSE)).

Q: [26 - Compiler Optimizations] What happens if you compile without -mpreferred-stack-boundary=2?
A: If you compile without -mpreferred-stack-boundary=2, the resulting code will allocate 16 bytes at a time, even for smaller data types.

Q: [27 - Inspecting the Stack with gdb] How can you inspect the stack with gdb?
A: You can inspect the stack with gdb by using commands such as "disassemble foo" to dump the assembler code for a function and "x / 12 wx" to examine memory.

Q: [28 - Function Body] What does the function body of foo() do?
A: The function body of foo() initializes a local variable c to 14, performs the calculation c = (a + b) * c, and returns the result.

Q: [29 - Function Epilogue] What does the function epilogue of foo() do?
A: The function epilogue of foo() restores the caller's frame on the stack and returns control to the original function caller using the leave and ret instructions.

Q: [30 - Ret Instruction] What does the ret instruction do?
A: The ret instruction pops the saved EIP from the stack and jumps to that address, returning control to the caller.

Q: [31 - Stack Smashing Control] How does changing the EIP allow control of the control flow?
A: Changing the EIP allows control of the control flow because it determines where the flow will jump next, potentially to arbitrary code.

Q: [32 - Valid Memory Location] What is required for a valid memory location in stack smashing?
A: A valid memory location in stack smashing must contain or be filled with valid executable machine code.

Q: [33 - Environment Variables Injection] How can environment variables be injected into a process?
A: Environment variables can be injected into a process by the command line interpreter, allowing them to be accessed and executed within the process.

Q: [34 - Built-in Functions Exploitation] How can built-in functions be exploited?
A: Built-in functions can be exploited by jumping to them and manipulating their execution, such as skipping authentication checks.

Q: [35 - Memory Control Exploitation] How can memory that can be controlled be exploited?
A: Memory that can be controlled can be exploited by filling it with executable code and jumping to it, allowing the execution of arbitrary code.

Q: [36 - Buffer Exploitation] How can the buffer itself be exploited?
A: The buffer itself can be exploited by injecting code into it and causing a buffer overflow to execute the injected code.

Q: [37 - Personalized Parameters Exploitation] How can personalized parameters be exploited?
A: Personalized parameters can be exploited by adding them to the stack and having the called function use them, such as calling exec() with our code as a parameter.

Q: [38 - Gets Function Vulnerability] Why is the gets function vulnerable?
A: The gets function is vulnerable because it reads from standard input without checking the maximum length, allowing buffer overflow attacks.

Q: [39 - Compiler Optimizations Impact] What impact do compiler optimizations have on stack allocation?
A: Compiler optimizations can impact stack allocation by aligning the stack to 16 bytes, even for smaller data types, if -mpreferred-stack-boundary=2 is not used.

Q: [40 - Inspecting the Stack with gdb Commands] What gdb commands can be used to inspect the stack?
A: gdb commands such as "disassemble foo" and "x / 12 wx" can be used to inspect the stack and examine memory.

Q: [01 - NOP Sled Explained] What is a NOP sled?
A: A NOP sled is a sequence of NOP (No Operation) instructions placed at the beginning of a buffer to ensure that wherever the execution jumps within this range, it will find a valid instruction and eventually reach the executable code.

Q: [01 - NOP Sled Explained] What does the NOP instruction do?
A: The NOP instruction (opcode on x86) does nothing and is used to create a safe landing zone for execution jumps in buffer overflow exploits.

Q: [02 - Where to Jump] Where should the execution jump in a buffer overflow exploit?
A: The execution should jump anywhere within the NOP sled range to ensure it reaches the executable code.

Q: [03 - What to Execute? Shellcode] What is the historical goal of an attacker in a buffer overflow exploit?
A: The historical goal of an attacker is to spawn a privileged shell on a local or remote machine.

Q: [03 - What to Execute? Shellcode] What is shellcode?
A: Shellcode is a sequence of machine instructions designed to perform a specific task, such as opening a shell, establishing a TCP connection, or launching a reverse shell.

Q: [03 - What to Execute? Shellcode] What is a common function used in shellcode to execute a shell?
A: The common function used is execve("/bin/sh").

Q: [04 - man execve] What does the execve function do?
A: The execve function executes the program referred to by pathname with a vector of arguments and a pointer to the environment.

Q: [04 - man execve] What does the execve function manage to switch context from user mode to kernel mode?
A: The execve function makes a system call, which switches the CPU from user mode to kernel mode to perform privileged operations.

Q: [05 - Calling convention] How is a system call invoked in Linux?
A: A system call is invoked by executing a software interrupt through the int instruction with the value (or equivalent instructions on modern processors).

Q: [05 - Calling convention] What are the steps to invoke a system call in Linux?
A: 1. Move the system call number into the EAX register. 2. Prepare arguments in the EBX, ECX, and EDX registers. 3. Execute the int instruction to switch to kernel mode.

Q: [06 - Writing shellcode] What are the steps to write shellcode?
A: 1. Write high-level code. 2. Compile and disassemble the code. 3. Analyze the assembly code. 4. Clean up the code. 5. Extract the opcode. 6. Create the shellcode.

Q: [07 - Simple x86 Shellcode Example] What does the simple x86 shellcode example demonstrate?
A: The example demonstrates how to execute execve("/bin/sh") by preparing the arguments and making the system call.

Q: [08 - Disassemble execve] What does the disassembled execve code show?
A: The disassembled execve code shows the assembly instructions for moving arguments into registers and invoking the system call.

Q: [09 - Memory Preparation] What needs to be prepared in memory for the execve shellcode?
A: The string "/bin/sh" terminated by a null byte, the address of the string, argv[0], a NULL pointer, and the environment pointer need to be prepared in memory.

Q: [10 - Memory Preparation] What is the purpose of the call instruction trick in shellcode?
A: The call instruction trick is used to determine the exact address of the string "/bin/sh" on the stack by pushing the return address onto the stack.

Q: [11 - Jump and Call Trick] How does the jump and call trick work?
A: The jump instruction jumps to a label, and the call instruction pushes the return address onto the stack, allowing the shellcode to determine its location in memory.

Q: [12 - Resulting Shellcode] What is the purpose of the resulting shellcode?
A: The resulting shellcode executes execve("/bin/sh") to spawn a shell, using the jump and call trick to determine the address of the string "/bin/sh".

Q: [13 - Zero Problems] What issue arises with zeros in shellcode?
A: Zeros in shellcode can cause problems because they are interpreted as string terminators, stopping any string-related operations.

Q: [14 - Substitutions] How can zeros be substituted in shellcode?
A: Zeros can be substituted with semantically equivalent instructions that do not contain zeros, such as using xor instructions instead of mov instructions.

Q: [15 - Resulting Shellcode (reprise)] What is the final shellcode after substituting zeros?
A: The final shellcode is a sequence of instructions that avoids zeros by using alternative instructions like xor and lea to achieve the same functionality.

Q: [16 - Shellcode, Ready to Use] What is the shellcode ready to use?
A: The shellcode ready to use is a binary string that can be injected into a vulnerable program to execute execve("/bin/sh") and spawn a shell.

Q: [17 - Preparing the Memory in Practice] What needs to be done to prepare the memory in practice for a buffer overflow exploit?
A: The buffer needs to be filled with the shellcode, the jump address, and NOPs to ensure the execution jumps to the shellcode.

Q: [18 - Memory That We Can Control] What are the pros and cons of exploiting memory that we can control?
A: Pros: Can be done remotely. Cons: The buffer may not be large enough, memory must be marked as executable, and the address must be guessed reliably.
